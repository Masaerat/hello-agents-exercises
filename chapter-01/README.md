# 第一章课后习题：初识智能体

## 习题 1.1：判断智能体类型

请分析以下四个案例中的主体是否属于智能体。如果属于，请从第一章介绍的一个或多个分类维度说明它属于哪类智能体，并给出理由。

**案例：**

- Case A：一台符合冯·诺依曼结构的超级计算机，拥有高达每秒 2 EFLOPS 的峰值算力。
- Case B：特斯拉自动驾驶系统在高速公路上行驶时，突然检测到前方有障碍物，需要在毫秒级做出刹车或变道决策。
- Case C：AlphaGo 在与人类棋手对弈时，需要评估当前局面并规划未来数十步的最优策略。
- Case D：ChatGPT 扮演的智能客服在处理用户投诉时，需要查询订单信息、分析问题原因、提供解决方案并安抚用户情绪。

### 参考答案

| 案例 | 是否属于智能体 | 可能类型 | 理由 |
| --- | --- | --- | --- |
| Case A：超级计算机 | 不一定属于 | 单独看不是智能体 | 算力强不等于具备智能体属性。智能体需要能够感知环境、基于目标自主决策，并通过行动影响环境。超级计算机可以运行智能体程序，但硬件本身只是计算平台。 |
| Case B：自动驾驶系统 | 属于 | 反应式智能体、基于模型的智能体、混合式智能体 | 自动驾驶系统通过摄像头、雷达、激光雷达等传感器感知道路环境，并通过刹车、转向、加速等执行器采取行动。障碍物避让强调毫秒级响应，因此具有强反应式特征；同时它也会维护车辆、车道和障碍物的内部状态模型，因此通常不是简单反射智能体，而是混合式系统。 |
| Case C：AlphaGo | 属于 | 基于目标、基于效用、学习型、规划式智能体 | AlphaGo 观察棋盘状态，评估候选落子，并选择行动。它的目标是赢棋，因此是目标导向的；它会评估局面价值和胜率，因此具有效用评估；它通过数据和自我博弈学习策略，因此也是学习型智能体；它会向未来多步展开搜索，因此属于规划式智能体。 |
| Case D：智能客服 | 属于，但前提是接入工具和业务环境 | LLM 驱动智能体、目标导向智能体、混合式智能体 | 如果 ChatGPT 只是被动聊天，它更像对话模型；但题目中它能够查询订单、分析投诉、提出方案并安抚用户，说明它能够通过用户消息和业务 API 感知环境，通过回复和工具调用采取行动，并围绕“解决投诉”这一目标进行推理，因此可视为 LLM 驱动的智能体。 |

## 习题 1.2：为智能健身教练设计 PEAS 模型

假设你需要设计一个“智能健身教练”。这个智能体能够：

- 通过可穿戴设备监测用户心率、运动强度等生理数据。
- 根据用户的健身目标（减脂、增肌、提升耐力等）动态调整训练计划。
- 在用户运动过程中提供实时语音指导和动作纠正。
- 评估训练效果并给出饮食建议。

请使用 PEAS 模型完整描述这个智能体的任务环境，并分析该环境具有什么特性。

### 参考答案

| PEAS 项 | 参考描述 |
| --- | --- |
| Performance 性能度量 | 训练安全性、目标达成度、运动效果、计划个性化程度、用户坚持率、伤病预防、用户满意度、长期健康改善、隐私保护。 |
| Environment 环境 | 用户身体状态、家庭或健身房空间、运动器械、可穿戴设备、用户时间安排、饮食习惯、健康史、室外运动时的天气、可能参与的教练或医生。 |
| Actuators 执行器 | 语音指导、App 提醒、训练计划调整、计时器、动作纠正提示、饮食建议、风险告警、暂停训练建议、与健身设备联动。 |
| Sensors 传感器 | 心率、运动强度、加速度计、陀螺仪、摄像头或姿态识别数据、用户反馈、训练日志、睡眠数据、体脂体重数据、饮食记录、用户输入的健康限制。 |

环境特性：

- 部分可观察：智能体无法直接知道用户的全部身体状态、疲劳程度、疼痛感、真实动机或完整动作质量，只能通过传感器和用户反馈间接判断。
- 随机性：同样的训练计划对不同用户、不同睡眠和饮食状态下的效果不同，用户也可能临时中断或动作变形。
- 序贯性：今天的训练会影响明天的疲劳、恢复、伤病风险和后续计划。
- 动态性：用户身体状态在运动中持续变化，智能体需要实时调整强度和提示。
- 连续性：心率、速度、关节角度、运动强度等信号通常是连续数值。
- 多智能体性：在部分场景下，用户、真人教练、医生、其他健康 App 或设备算法都会影响环境。

## 习题 1.3：电商退款审核中的 Workflow 与 Agent

某电商公司正在考虑两种方案来处理售后退款申请。

**方案 A：Workflow**

- 对于 7 天内的一般商品：
  - 金额小于 100 RMB 自动通过。
  - 金额在 100-500 RMB 之间由客服审核。
  - 金额大于 500 RMB 需主管审批。
- 对于特殊商品（如定制品），默认拒绝退款。
- 对于超过 7 天的商品，无论金额大小，都只能由客服或主管审核。

**方案 B：Agent**

搭建一个智能体系统，让它理解退款政策、分析用户历史行为、评估商品状况，并自主决策是否批准退款。

请分析两种方案各自的优缺点、适用场景，以及是否存在结合二者优点的方案 C。

### 参考答案

Workflow 的优点是规则明确、行为稳定、可预测，容易审计和解释，适合合规要求较高的流程；对高频、重复、边界清晰的申请处理效率高，实现和维护成本也相对可控。缺点是面对复杂或模糊情况时不够灵活，规则变多后容易臃肿，难以理解自然语言投诉、图片证据、上下文情绪等非结构化信息，也可能让合理但不符合固定规则的特殊案例获得较差体验。

Agent 的优点是能理解自然语言投诉，并综合用户历史、商品状态、客服记录、风控信号等上下文；适合处理规则边界模糊、证据复杂、需要解释和沟通的场景，也可以主动追问缺失信息并生成更个性化的处理方案。缺点是可能产生幻觉或不一致决策，如果缺少日志和约束，审计难度更高，也可能错误理解政策或越权处理高风险退款。

Workflow 更适合政策稳定、边界清晰、判断条件结构化、风险较低或能被金额和品类规则清楚界定的场景。Agent 更适合用户描述复杂、证据非结构化、需要综合判断和高质量沟通的场景。

更稳妥的方案 C 是“规则兜底 + 智能体处理复杂案例”：对明显符合规则的低风险申请使用 Workflow 自动通过或拒绝；对高金额、特殊商品、疑似欺诈、证据复杂的申请交给 Agent 汇总信息并给出建议；高风险或敏感案例保留人工审批。Agent 的工具调用、证据来源、判断过程和最终建议都应记录，且法律要求、黑名单、金额上限等硬性规则不能被 Agent 绕过。

## 习题 1.4：扩展智能旅行助手

在第一章 1.3 节的智能旅行助手基础上，思考如何加入以下能力。你可以只描述设计思路，也可以进一步写出伪代码或代码。

**新增能力：**

- 添加“记忆”功能，让智能体记住用户偏好，例如喜欢历史文化景点、预算范围、是否偏好室内活动等。
- 当推荐景点门票已售罄时，智能体能够自动推荐备选方案。
- 如果用户连续拒绝 3 个推荐，智能体能够反思并调整推荐策略。

### 参考答案

可以把记忆分为两类：用户长期偏好和当前会话状态。长期偏好包括喜欢历史文化景点、预算范围、是否偏好室内活动、是否愿意长距离步行、饮食偏好等；当前会话状态包括本轮已推荐景点、被拒绝的选项、拒绝原因、当前城市、天气和门票状态等。

循环可以调整为：

```text
Observation -> 更新会话状态/记忆 -> Thought -> 选择工具 -> Action -> Observation
```

系统提示词中应要求智能体在推荐前读取用户偏好，并在用户反馈后更新记忆。长期稳定偏好可以写入长期记忆；临时信息如“今天下雨”“本轮推荐过颐和园”更适合保留在会话状态中。

处理门票售罄时，需要新增工具：

```text
check_ticket(attraction, date) -> available | sold_out | unknown
search_alternative_attractions(city, constraints) -> list
```

当 `check_ticket` 返回 `sold_out` 时，智能体不应直接结束，而应把售罄结果作为新的 Observation，重新规划：

```text
Thought: 当前景点门票已售罄，需要基于用户偏好、天气和预算搜索替代方案。
Action: search_alternative_attractions(city="北京", constraints="历史文化, 室外, 预算适中")
```

连续拒绝时，需要记录拒绝次数和原因：

```text
state.rejected_recommendations = [
  { attraction: "A", reason: "太贵" },
  { attraction: "B", reason: "太远" },
  { attraction: "C", reason: "不感兴趣" }
]
```

当连续拒绝达到 3 次时，智能体应进入反思或澄清步骤：

```text
Thought: 用户连续拒绝了 3 个推荐，说明我可能误解了偏好或约束。应先澄清关键偏好，而不是继续盲目推荐。
Action: ask_user(question="你更希望我优先考虑价格、距离、室内活动，还是人少安静的景点？")
```

同时应保留最大循环次数，避免无限重试；当用户接受推荐、缺少关键信息需要追问、没有可靠候选项或达到最大循环次数时停止。门票状态不能编造，如果工具返回 unknown，应明确说明不确定并建议用户确认。

## 习题 1.5：智能体中的系统 1 与系统 2

卡尼曼的“系统 1”（快速直觉）和“系统 2”（慢速推理）理论为神经符号主义 AI 提供了很好的类比。

请先构思一个具体的智能体落地应用场景，例如医疗诊断助手、法律咨询机器人、金融风控系统、智能客服等，并说明哪些任务适合由系统 1 处理，哪些任务适合由系统 2 处理，以及两个系统如何协同工作。

### 参考答案

以下以“金融风控智能体”为例。

适合系统 1 处理的任务包括：快速识别常见欺诈模式，对明显低风险或高风险交易进行初步打分，识别异常设备、异常登录地点、异常金额、异常交易频率，并在毫秒级给出风险预警。这些任务强调快速模式识别和实时响应。

适合系统 2 处理的任务包括：分析多个风险信号相互矛盾的复杂案例，检查明确的合规规则和业务政策，解释为什么拦截、放行或要求二次验证，在放行、延迟、拦截、人工审核之间权衡，并对用户申诉或事后审计进行推理。这些任务需要慢速推理、规则检查、可解释性和责任边界。

两者可以这样协同：系统 1 先对交易进行快速风险评分，并标出可疑特征；系统 2 对中高风险或高影响交易进行规则核查和上下文推理；智能体根据综合判断选择放行、二次验证、拦截或转人工；事后根据真实欺诈、误杀和用户申诉结果，更新模型和规则。

医疗诊断助手、法律咨询机器人和智能客服也可以采用类似思路：系统 1 负责快速识别模式或召回候选信息，系统 2 负责规则化推理、约束检查、解释和最终决策。

## 习题 1.6：LLM 智能体的局限与评估

尽管大语言模型驱动的智能体系统展现出了强大能力，但它们仍然存在诸多局限。

请分析：

- 为什么智能体有时会产生“幻觉”，生成看似合理但实际错误的信息？
- 在 1.3 节的案例中，最大循环次数被设置为 5 次。如果没有这个限制，智能体可能会陷入什么问题？
- 应该如何评估一个智能体的“智能”程度？仅使用准确率指标是否足够？

### 参考答案

智能体产生幻觉，首先是因为大语言模型本质上是在生成概率上合理的文本，而不是直接保证事实正确。模型可能缺少最新、私有或专业领域信息；用户提示不明确时，模型可能用看似合理的假设补全缺失信息；工具返回的信息也可能缺失、过期、噪声较多，或被智能体错误解释。多轮循环中，小错误还可能在后续步骤中不断累积。如果缺少外部验证，智能体可能过度相信自己的中间推理。

如果取消最大循环次数，智能体可能陷入无限循环，反复调用同一个工具或重复失败计划；也可能造成 token、API 调用和外部服务费用持续增加；如果行动有副作用，还可能重复下单、重复发消息或重复写数据库。长循环还会带来目标漂移、错误放大和用户体验变差等问题。因此最大循环次数、超时时间和预算上限是必要的工程安全机制。

评估智能体不能只看准确率。智能体会与环境交互、使用工具、进行多步规划，因此评价维度应包括任务成功率、事实可靠性、鲁棒性、效率、安全性、可解释性、适应性、用户满意度和长程可靠性。准确率只能衡量某类输出是否正确，无法完整反映智能体是否安全、可靠、可控地完成了真实任务。
