# 第五章课后习题：基于低代码平台的智能体搭建

## 习题 5.1：低代码平台定位与开发模式选择

本章介绍了四个各具特色的低代码平台：`Coze`、`Dify`、`FastGPT` 和 `n8n`。请分析：

- 这四个平台在核心定位和设计理念上有什么区别？它们分别解决了智能体开发中的哪些痛点？
- 低代码平台与纯代码开发各有优劣，此外，也有部分功能用平台实现，部分功能用代码实现的“混合开发”模式。思考三种开发模式分别适合哪些场景？请举例说明。

### 参考答案

`Coze` 的核心定位是面向业务人员和轻量开发者的智能体搭建平台，强调低门槛、插件丰富、发布渠道友好。它解决的主要痛点是“不会写代码也能快速搭建一个可用智能体”，适合做客服助手、内容生成助手、信息查询助手等快速验证型应用。

`Dify` 更偏向企业级 LLM 应用开发平台，强调工作流、知识库、工具调用、模型管理、本地部署和应用运维。它解决的痛点是企业要把大模型能力接入真实业务流程时，需要可配置、可观测、可部署、可迭代的完整平台。相比 Coze，Dify 更适合复杂业务应用和私有化场景。

`FastGPT` 的核心优势在知识库问答和 RAG 应用搭建，设计理念是让用户快速把文档、知识库、工作流和模型能力组合成企业知识助手。它解决的痛点是“如何快速把企业文档变成可问答、可追溯的知识系统”，适合内部制度问答、产品文档问答、售前知识库等场景。

`n8n` 本质上是自动化工作流平台，智能体能力是其工作流生态中的一部分。它强调连接各种 SaaS、数据库、邮件、Webhook、脚本和 AI 节点，解决的痛点是跨系统自动化编排。它适合“收到事件后自动触发多个系统操作”的场景，例如邮件处理、CRM 同步、工单分派、订单流转。

纯低代码开发适合需求清晰、流程相对标准、上线速度优先、团队工程能力有限的场景。例如一个产品经理要在一周内做出“每日行业新闻摘要助手”，用 Coze 或 Dify 会比从零开发更快。

纯代码开发适合对性能、权限、安全、复杂业务逻辑、深度定制和可测试性要求很高的场景。例如金融风控审批、医疗辅助诊断、企业级代码审查平台等，通常需要完整的工程架构、严格审计、灰度发布、监控告警和自定义模型调用策略。

混合开发适合既要快速上线，又有一部分能力需要深度定制的场景。常见做法是用 Dify、FastGPT 或 n8n 搭建知识库、工作流和后台管理界面，把高风险或强业务逻辑部分封装成内部 API、MCP Server 或自定义插件。例如企业合同审核系统可以用 Dify 编排“上传合同 -> 检索条款库 -> 调用审核模型 -> 生成人工复核报告”的流程，但合同解析、权限校验、审计日志和 OA 集成由后端代码实现。

## 习题 5.2：扩展 Coze“每日 AI 简报”智能体

在5.2节的 `Coze` 案例中，我们构建了一个“每日AI简报”智能体。请基于此案例进行扩展思考：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 当前的简报生成是被动触发的（用户主动询问）。如何改造这个智能体，使其能够每天早上8点自动生成简报并推送到指定的飞书群或微信公众号？
- 简报的质量高度依赖于提示词设计。请尝试优化5.2.2节中的提示词，使生成的简报更加专业、结构更清晰，或者增加“热点分析”、“趋势预测”等新功能。
- `Coze` 当前不支持 `MCP` 协议被认为是一个重要局限（在习题的写作过程中，`feature-mcp` 虽然在 [`Coze Studio Q4 2025 Product Roadmap`](https://github.com/coze-dev/coze-studio/issues/2218) 中了，但是还尚未实现）。请简述，什么是 `MCP` 协议？它为什么重要？如果 `Coze` 未来支持 `MCP`，会带来哪些新的可能性？

### 参考答案

要把“每日 AI 简报”从被动触发改造成定时推送，可以采用“定时任务 + 简报生成智能体 + 推送渠道”的架构。第一步在 Coze 或外部平台中配置定时触发器，每天早上 8 点触发一次任务；如果 Coze 当前能力不足，可以用飞书机器人定时任务、云函数、GitHub Actions、n8n Cron 节点或服务器上的 crontab 调用 Coze API。第二步由智能体调用新闻搜索、RSS、行业资讯 API 或指定网站抓取工具，获取最近 24 小时的 AI 新闻。第三步让模型按固定结构生成简报。第四步调用飞书群机器人 Webhook、微信公众号草稿接口或企业微信机器人接口推送。

一个可执行流程如下：

```text
Cron 每天 08:00 触发
-> 调用新闻/RSS/搜索工具获取候选新闻
-> 去重、按热度和相关性筛选 5-8 条
-> 调用 Coze 智能体生成简报
-> 调用飞书群机器人 Webhook 或微信公众号接口
-> 记录推送日志和失败重试状态
```

优化后的提示词可以设计为：

```text
你是一名专业的 AI 行业研究员，请基于检索到的新闻材料生成“每日 AI 简报”。

要求：
1. 只基于输入材料总结，不编造不存在的公司、产品、融资、论文或发布日期。
2. 输出结构固定为：
   - 今日摘要：用 3 句话概括今天 AI 行业最重要变化。
   - 重点新闻：列出 5 条，每条包含标题、来源、时间、核心事实、影响分析。
   - 热点分析：指出今天新闻中反复出现的 1-2 个主题，并解释原因。
   - 趋势预测：基于已有事实给出短期趋势判断，必须标注“不确定性”。
   - 行动建议：分别给产品、研发、运营三个角色提出一条建议。
3. 语气专业、简洁，避免营销口吻。
4. 如果材料证据不足，请明确写“材料不足，无法判断”。
```

`MCP` 即 Model Context Protocol，是一种用于连接大模型应用与外部工具、资源、提示模板的开放协议。它采用客户端-服务器架构，MCP Host 通过 MCP Client 连接一个或多个 MCP Server；Server 可以暴露工具、资源和提示词，工具调用使用结构化参数，底层协议基于 JSON-RPC，并支持工具发现、能力协商和通知机制。

MCP 重要的原因在于它把“智能体如何接入外部系统”标准化了。没有 MCP 时，每个平台都要为同一个外部系统单独开发插件，模型、平台、工具之间形成大量重复适配。支持 MCP 后，一个高德地图 MCP Server、企业知识库 MCP Server 或数据库 MCP Server 可以被多个智能体平台复用，降低集成成本，也提升工具调用的可移植性和可治理性。

如果 Coze 未来支持 MCP，会带来三类新可能。第一，企业可以把内部系统封装成 MCP Server，让 Coze 智能体安全访问 CRM、ERP、知识库和工单系统。第二，开发者可以复用社区 MCP Server，不必等待 Coze 插件商店逐一适配。第三，智能体可以动态发现工具和资源，使复杂业务智能体更容易扩展。但同时也要配套权限控制、审计日志、工具白名单和敏感数据过滤，否则 MCP 会扩大工具误用和数据泄露风险。

## 习题 5.3：分析 Dify“超级智能体个人助手”

在5.3节的 `Dify` 案例中，我们构建了一个功能全面的“超级智能体个人助手”。请深入分析：

- 案例中使用了“问题分类器”进行智能路由，将不同类型的请求分发到不同的子智能体。这种多智能体架构有什么优势？如果不使用分类器，而是让一个单一的智能体处理所有任务，会遇到什么问题？
- 数据查询模块需要为大模型提供清晰的表结构信息。如果数据库有50张表、每张表有20个字段，直接将所有 `DDL` 语句放入提示词会导致上下文过长。请设计一个更智能的方案来解决这个问题。
- `Dify` 支持本地部署和云端部署两种模式。请对比这两种模式在数据安全、成本、性能、维护难度等方面的差异，并说明各自适用的场景。

### 参考答案

问题分类器的作用是先判断用户请求属于哪一类任务，再把请求分发给最合适的子智能体或工作流。这样做的优势是职责清晰、提示词更短、工具权限更可控、效果更容易优化。例如日程管理智能体只需要日历工具和日程规则，数据查询智能体只需要数据库 schema 和 SQL 生成规则，文案优化智能体只需要写作风格要求。每个子智能体的上下文更聚焦，错误率通常会低于一个“大而全”的智能体。

如果用单一智能体处理所有任务，会遇到几个问题。第一，提示词会越来越长，模型难以稳定遵守所有规则。第二，工具数量过多，工具选择错误概率上升。第三，权限边界不清晰，例如文案任务不应该拥有数据库写权限。第四，调试困难，无法快速判断是分类、工具、提示词还是子任务逻辑出错。第五，后续扩展时容易形成难以维护的“巨型智能体”。

对于 50 张表、每张表 20 个字段的数据库，不应把所有 DDL 直接塞进提示词。更合理的方案是构建“分层 schema 检索”。第一层维护数据库目录，包括表名、中文说明、业务域、关键字段和示例问题。第二层根据用户问题先检索最相关的 3-5 张表。第三层只把这些表的详细 DDL、字段含义、主外键关系和示例 SQL 放入提示词。第四层让模型生成 SQL 后，用 SQL Parser 或只读数据库权限进行校验，禁止危险语句。第五层执行失败时，把错误信息和相关 schema 反馈给模型修复。

示例流程：

```text
用户问题
-> 意图识别和业务域识别
-> 检索表目录，召回候选表
-> 检索字段说明、关系图、样例 SQL
-> 生成 SQL
-> 语法校验和权限校验
-> 只读执行
-> 结果解释
```

云端部署的优势是开箱即用、升级维护简单、初期成本低、适合快速试验。缺点是数据要进入云服务或第三方模型链路，敏感数据、合规要求和网络延迟可能成为问题。它适合个人应用、原型验证、公开数据问答、非敏感业务助手。

本地部署的优势是数据可留在私有环境，便于接入内网系统、做权限隔离、审计和合规控制，也可以根据业务需要定制模型和插件。缺点是需要自己维护服务器、数据库、向量库、对象存储、模型网关、备份和升级，初期运维成本更高。它适合金融、政务、法律、医疗、制造业等敏感场景，也适合需要深度集成企业 OA、CRM、ERP、文档管理系统的应用。

## 习题 5.4：扩展 n8n“智能邮件助手”

在5.5节的 `n8n` 案例中，我们构建了一个“智能邮件助手”。请思考以下问题：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 案例中使用的 `Simple Vector Store` 和 `Simple Memory` 都是基于内存的，服务重启后数据会丢失。请查阅 `n8n` 文档，尝试将其替换为持久化存储方案（如 `Pinecone`、`Redis` 等），并说明配置过程。
- 当前的邮件助手只能处理文本邮件。如果用户发送的邮件中包含附件（如 `PDF` 文档、图片），你会如何扩展这个工作流，使智能体能够理解附件内容并做出相应回复？
- `n8n` 的核心优势在于“连接”能力。请设计一个更复杂的自动化场景：当客户在电商平台下单后，自动触发一系列操作（发送确认邮件、更新库存数据库、通知物流系统、在 `CRM` 中记录客户信息）。请画出工作流的节点连接图并说明关键配置。

### 参考答案

替换 `Simple Vector Store` 时，可以使用 `Pinecone Vector Store`、Qdrant、Supabase Vector Store、Postgres PGVector 等持久化向量存储。以 Pinecone 为例，先在 Pinecone 创建 index，确定向量维度与嵌入模型一致；然后在 n8n 中配置 Pinecone 凭据；在工作流中把原来的 `Simple Vector Store` 替换为 `Pinecone Vector Store` 节点；写入知识时选择 insert/upsert 模式，查询时选择 retrieve 模式；最后把它连接到嵌入模型节点和 AI Agent/Retriever 节点。这样邮件知识库和历史资料不会因为 n8n 重启而丢失。

对话记忆可以替换为 Redis、Postgres 或外部数据库。配置思路是为每个用户或邮箱生成稳定的 `session_id`，例如 `email:{sender_address}`，把对话历史写入 Redis Chat Memory 或数据库表。字段至少包括用户标识、消息角色、消息内容、时间戳、会话 ID。回复时按会话 ID 取最近若干轮历史，既保留上下文，又避免提示词无限增长。

处理附件时，需要在邮件触发节点后增加附件分支。流程如下：

```text
Email Trigger
-> IF 是否包含附件
   -> 无附件：直接进入文本清洗和 AI Agent
   -> 有附件：Split In Batches 遍历附件
      -> PDF：Extract PDF Text / MarkItDown / 外部解析 API
      -> 图片：OCR 或多模态模型识别
      -> Word/Excel：文档解析节点或自定义 Code 节点
      -> 文本归一化
      -> 分块和向量化
      -> 合并邮件正文与附件摘要
      -> AI Agent 生成回复
-> 人工确认或自动发送
```

对于 PDF，需要提取正文、表格和关键页码；对于图片，需要 OCR 出文本，必要时调用视觉模型理解截图、票据或合同图片；对于大附件，应先生成摘要，再让智能体基于摘要和原文片段回答，避免上下文过长。

电商下单自动化场景可以设计为：

```text
Webhook / Shopify Trigger / 电商平台订单触发
-> Validate Order 校验订单字段和签名
-> IF 支付状态是否成功
   -> 否：记录异常并通知运营
   -> 是：
      -> Send Email 发送订单确认邮件
      -> Database 更新库存
      -> IF 库存是否低于阈值
         -> 通知采购或创建补货任务
      -> HTTP Request 调用物流系统创建运单
      -> CRM Node 创建或更新客户档案
      -> Slack/飞书通知内部群
      -> Log 写入审计记录
```

关键配置包括：Webhook 需要校验签名，防止伪造订单；库存更新必须使用事务或幂等订单号，避免重复扣减；物流系统调用要设置失败重试和异常分支；CRM 记录要用邮箱或手机号做去重；邮件发送前要使用模板变量填充订单号、商品、金额、预计发货时间；所有外部 API 凭据应放在 n8n Credentials 中，不应硬编码在节点参数里。

## 习题 5.5：低代码平台中的提示词工程

提示词工程在低代码平台中同样至关重要。本章展示了多个平台的提示词设计案例。请分析：

- 对比5.2.2节（`Coze`）、5.3.2节（`Dify`）和5.5.4节（`n8n`）中的提示词设计，它们在结构、风格和侧重点上有什么不同？这些差异是否与平台特性相关？
- 在 `Dify` 的“文案优化模块”中，提示词要求输出“超过500字”。这种对输出长度的硬性要求是否合理？在什么情况下应该限制输出长度，什么情况下应该让模型自由发挥？

### 参考答案

Coze 案例中的提示词通常更偏“角色 + 任务 + 输出格式”，重点是让一个面向用户的智能体表现出稳定的人设和能力。它适合快速配置对话式助手，因此提示词会强调回答风格、信息来源、简报结构和用户体验。

Dify 案例中的提示词更偏工作流模块化设计。不同节点或子智能体承担不同任务，例如问题分类、数据查询、文案优化、工具调用。提示词通常会更明确地规定输入变量、处理规则、输出字段和异常情况。这与 Dify 的工作流能力相关：每个节点都应有清晰输入输出，方便后续节点消费。

n8n 案例中的提示词更贴近自动化流程。它通常要和邮件触发器、数据库、向量库、外部 API 节点配合，因此重点不是完整对话人设，而是让 AI 节点在某个自动化步骤中完成明确动作，例如总结邮件、判断意图、生成回复草稿或提取结构化字段。它的提示词更强调可执行性、结构化输出和与上下游节点的字段匹配。

要求输出“超过 500 字”不一定合理。它的好处是可以避免模型输出过短，适合长文案、营销文章、报告、方案说明等需要一定展开度的场景。但硬性字数也可能导致模型填充废话，降低信息密度，甚至为了凑字数引入不必要内容。

应该限制长度的场景包括：短信、邮件标题、客服回复、摘要、移动端卡片、API 字段、广告语、数据库字段、法律免责声明等。这些场景对可读性、版面和成本敏感，适合规定“100 字以内”“3 个要点”“每条不超过 30 字”。

应该让模型相对自由发挥的场景包括：头脑风暴、初稿创作、复杂方案分析、深度报告、教学解释等。这类任务更看重完整性和洞察力，可以改用“覆盖哪些要点”“结构必须包含哪些部分”“不要遗漏哪些信息”来约束，而不是只用字数约束。更好的做法是规定结构和质量标准，例如“至少包含背景、问题、方案、风险、下一步”，而不是单纯要求超过 500 字。

## 习题 5.6：工具、插件与 MCP 集成

工具和插件是低代码平台的核心能力扩展方式。请思考：

- `Coze` 拥有丰富的插件商店，`Dify` 拥有8000+的插件市场，`FastGPT` 原生支持 MCP 协议，`n8n` 拥有数百个预置节点。如果这三个平台都没有你需要的某个特定工具（如“连接公司内部系统的 `API`”），你会如何解决？
- 在5.3.2节中，我们使用了 `MCP` 协议集成了高德地图、饮食推荐等服务。请调研并说明：`MCP` 协议与传统的 `RESTful API` 以及 `Tool Calling` 有哪些区别？为什么说 `MCP` 是智能体工具调用的“新标准”？
- 假设你要为 `Dify` 开发一个自定义插件，使其能够调用你公司的内部知识库系统。请查阅 `Dify` 的插件开发文档，概述开发流程和关键技术点。

### 参考答案

如果平台没有现成工具，优先方案是把公司内部系统封装成标准接口，再按平台能力接入。对于 Coze，可以开发自定义插件或通过 HTTP API 间接调用；对于 Dify，可以开发 Tool 插件、Extension 插件或在工作流中使用 HTTP Request；对于 FastGPT，可以优先封装为 MCP Server；对于 n8n，可以使用 HTTP Request、Code 节点、自定义节点或社区节点。无论哪种方式，都应该把认证、限流、审计、错误处理和权限控制放在服务端，而不是完全交给提示词约束。

`RESTful API` 是通用 Web API 设计风格，关注资源、HTTP 方法、状态码和请求响应。它本身不关心大模型如何发现工具、如何理解参数、如何把结果作为上下文使用。开发者需要额外写 OpenAPI 描述、SDK 或适配层。

`Tool Calling` 通常是模型厂商或框架提供的函数调用能力。开发者把函数名、描述和 JSON Schema 提供给模型，模型生成结构化调用参数，由应用执行函数。它解决了模型“调用函数”的问题，但往往绑定某个模型供应商或某个框架，跨平台复用能力有限。

`MCP` 更像是面向智能体的工具和上下文连接协议。它不只是函数列表，还定义了 Host、Client、Server 的架构，支持工具、资源、提示模板等原语，支持能力协商、工具发现、工具调用、通知和多种传输方式。MCP Server 可以独立于具体模型和平台运行，让不同客户端复用同一套工具连接。

因此，MCP 被称为智能体工具调用的新标准，是因为它解决了“每个平台都要重复适配每个工具”的问题。它让工具提供方只需实现 MCP Server，智能体平台只需实现 MCP Client，就可以复用工具、资源和提示模板。但它不是 REST 的替代品。实际工程中，MCP Server 内部仍然可能调用 REST、GraphQL、数据库或消息队列。

为 Dify 开发内部知识库插件，可以按以下流程进行：

1. 选择插件类型。如果目标是让 Agent 查询内部知识库，应选择 Tool 插件；如果要把内部系统作为知识管道数据源，可以选择 Datasource 插件；如果通过 Webhook 扩展外部服务，可选 Extension。
2. 安装 Dify Plugin CLI，并用脚手架创建插件项目。
3. 编写插件清单，声明插件名称、版本、作者、权限、配置项和凭据字段。
4. 定义工具接口，例如 `search_internal_kb(query, department, top_k)`，用 schema 描述参数类型、必填项和说明。
5. 在插件代码中实现认证、请求内部知识库 API、结果清洗、错误处理和超时重试。
6. 本地调试插件，确认 Dify 能识别工具、展示参数并成功调用。
7. 打包插件，上传到私有环境或发布到插件市场。
8. 在生产环境中配置最小权限 API Key、访问白名单、审计日志、速率限制和敏感信息脱敏。

关键技术点是结构化参数定义、凭据安全、错误返回格式、超时和重试、内部系统权限映射、结果摘要与引用来源，以及插件版本兼容。

## 习题 5.7：平台选型分析

平台选型是智能体产品成功的关键决策之一。假设你是一家初创公司的技术负责人，公司计划开发以下三个AI应用，请为每个应用选择最合适的平台（`Coze`、`Dify`、`FastGPT`、`n8n` 或纯代码开发），并详细说明理由：

<strong>应用A</strong>：面向C端用户的“AI写作助手”小程序，需要快速上线验证市场需求，预算有限，团队中只有1名前端工程师和1名产品经理。

<strong>应用B</strong>：面向企业客户的“智能合同审核系统”，需要处理敏感的法律文档，要求数据不能离开客户的私有环境，需要与客户现有的OA系统、文档管理系统深度集成。

<strong>应用C</strong>：内部使用的“研发效能提升工具”，需要自动化处理代码审查、测试报告生成、Bug跟踪、项目进度同步等多个研发流程环节，团队有较强的技术实力。

对于每个应用，请从以下维度（包括但不限于）进行分析：

> <strong>提示</strong>：平台能力是否满足需求，多快能上线，开发成本、运营成本，后续迭代的难度，未来功能扩展的空间

- 技术可行性
- 开发效率
- 成本控制
- 可维护性
- 可扩展性
- 数据安全与合规性

### 参考答案

应用 A 最适合优先选择 Coze，也可以用 Dify Cloud 做备选。原因是它面向 C 端，需要快速上线验证需求，团队工程资源很少。Coze 的对话式智能体、插件、发布渠道和低代码配置能让产品经理快速调整角色设定、写作模板和交互流程，前端工程师只需要负责小程序壳、登录、支付或历史记录等外围功能。

从技术可行性看，AI 写作助手主要依赖提示词、模板、少量工具和用户交互，低代码平台足够支撑。开发效率上，Coze 可以最快做出 MVP。成本方面，初期不需要维护复杂后端。可维护性方面，提示词和写作模板可以由产品人员迭代。可扩展性方面，如果后续要做会员体系、作品库、风格训练和私有模型，可以再迁移到混合架构。数据安全方面，普通 C 端写作数据敏感度相对低，但仍需隐私政策、数据删除和内容审核。

应用 B 最适合 Dify 本地部署 + 自定义插件 + 必要的纯代码后端，不建议只用云端低代码平台。合同审核涉及敏感法律文档，明确要求数据不能离开客户私有环境，还要和 OA、文档管理系统深度集成。Dify 本地部署可以提供工作流、知识库、模型接入和插件体系；合同解析、权限控制、审计日志、文档水印、脱敏、OA 集成等部分应由代码实现。

从技术可行性看，Dify 能承载“上传合同 -> 检索合同条款库 -> 生成风险点 -> 人工复核”的主流程。开发效率比纯代码更高。成本方面，私有化部署初期成本高，但能满足企业客户合规要求。可维护性方面，业务人员可以维护审核规则和知识库，工程团队维护插件和集成服务。可扩展性方面，后续可以加入条款比对、版本追踪、多模型复核。数据安全方面，本地部署、访问控制、日志审计和私有模型是关键要求。

应用 C 最适合 n8n + 纯代码服务的混合模式，也可以在核心智能逻辑上接入自建 Agent 框架。研发效能工具需要连接 GitHub/GitLab、CI、测试平台、Jira、飞书、项目管理系统等多个工具，n8n 的连接和自动化编排能力非常适合做流程触发和系统集成。例如代码合并后触发代码审查，测试失败后生成报告并创建 Bug，版本发布后同步项目进度。

从技术可行性看，n8n 能快速打通多个研发系统；复杂代码分析、AST 解析、安全扫描和测试报告分析应由纯代码服务实现。开发效率上，n8n 负责编排能显著减少胶水代码。成本方面，内部工具可以自托管 n8n，复用已有系统 API。可维护性方面，流程节点可视化，便于研发团队调整。可扩展性方面，技术团队强，可以逐步把关键能力沉淀成内部 API、MCP Server 或自定义 n8n 节点。数据安全方面，代码和缺陷信息敏感，应使用内网部署、最小权限 Token、审计日志和敏感信息脱敏。
