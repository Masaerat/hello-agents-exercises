# 第十二章课后习题：智能体性能评估

> <strong>提示</strong>：部分习题没有标准答案，重点在于培养学习者对智能体性能评估的综合理解和实践能力。

## 习题 12.1：评估基准、客服指标与评估挑战

本章介绍了多个智能体评估基准。请分析：

- 在 12.1.2 节中介绍了 BFCL、GAIA、AgentBench 等评估基准。请对比 BFCL 和 GAIA：它们分别评估智能体的哪些核心能力？为什么 BFCL 使用 AST 匹配算法，而 GAIA 使用准精确匹配（Quasi Exact Match）？这两种评估方法各有什么优缺点？
- 假设你要构建一个“智能客服系统”，需要评估以下能力：（1）理解用户意图的准确性；（2）调用后台 API 的正确性；（3）回答的友好性和专业性；（4）处理异常情况的鲁棒性。请为每个能力选择或设计合适的评估指标和方法。
- 在 12.1.1 节中提到，智能体评估面临“输出不确定性”、“评估标准多样性”、“评估成本高昂”三大挑战。请针对每个挑战提出具体的解决方案，并分析方案的可行性和局限性。

### 参考答案

BFCL 主要评估模型的函数调用能力，包括能否选择正确函数、生成正确参数、处理多个函数、并行函数和无关工具干扰。它关注的是“工具调用结构是否正确”。GAIA 主要评估通用智能体完成真实世界问题的综合能力，通常需要搜索、推理、文件处理、计算和多步操作。它关注的是“最终任务答案是否正确”。

BFCL 使用 AST 匹配，是因为函数调用有明确结构。字符串可能不同但语义等价，例如参数顺序不同、空格不同、引号不同，AST 能更稳健地比较函数名、参数名和参数值。优点是精确、自动化程度高、适合结构化调用；缺点是对语义等价但结构不同的调用仍可能误判，例如单位换算已提前完成但参数不同。

GAIA 使用准精确匹配，是因为最终答案可能有多种表达形式，例如“42”“42.0”“四十二”可能都正确。准精确匹配会做大小写、空格、标点、数值格式等归一化。优点是简单、成本低、可批量自动评估；缺点是对开放答案、解释型答案、部分正确答案和语义等价答案处理不足。

智能客服系统可以这样评估：

```text
意图理解准确性：
  指标：intent accuracy、macro F1、混淆矩阵。
  方法：人工标注一批真实客服问题，比较模型分类结果。

后台 API 调用正确性：
  指标：工具选择准确率、参数准确率、调用顺序准确率、越权调用率。
  方法：构造带标准工具调用轨迹的测试集，使用结构化匹配和沙箱执行验证。

回答友好性和专业性：
  指标：礼貌性、清晰度、政策一致性、事实准确性。
  方法：人工评分 + LLM Judge + 抽样复核。

异常处理鲁棒性：
  指标：缺失信息澄清率、API 失败恢复率、低置信转人工率、错误拒答率。
  方法：构造 API 超时、订单不存在、用户输入矛盾、恶意诱导等测试场景。
```

输出不确定性可以通过固定随机种子、降低温度、多次采样取平均、使用结构化输出和评估置信区间缓解。局限是实际生产仍有随机性，过低温度也可能降低创造性。

评估标准多样性可以通过分维度 rubrics 解决，例如事实、工具、语气、安全、效率分别评分，而不是只给总分。局限是维度设计和权重仍带有主观性，需要业务方确认。

评估成本高昂可以通过分层评估解决：日常用规则和小样本快速评估，发布前用标准评估集，重大版本用人工评审和线上灰度。局限是低成本评估可能漏掉复杂问题，高质量评估仍需要人工和真实数据。

## 习题 12.2：BFCL 与工具调用评估器扩展

BFCL（Berkeley Function Calling Leaderboard）是评估工具调用能力的重要基准。基于 12.2 节的内容，请深入思考：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 在 12.2.3 节的 AST 匹配算法中，我们通过比较抽象语法树来判断函数调用是否正确。请分析：为什么 AST 匹配比简单的字符串匹配更合适？在什么情况下 AST 匹配可能会产生误判（假阳性或假阴性）？如何改进 AST 匹配算法来提高准确性？
- BFCL 数据集包含 simple、multiple、parallel、irrelevance 四个类别。请为每个类别设计 2-3 个新的测试样本，要求能够测试智能体在该类别下的边界情况或容易出错的场景。
- 请基于 12.2.4 节的代码，扩展 BFCL 评估器，添加以下功能：（1）支持评估工具调用的执行顺序（对于有依赖关系的多个工具调用）；（2）评估工具调用的效率（如是否使用了最少的调用次数）；（3）生成详细的错误分析报告（如哪些类型的错误最常见）。

### 参考答案

AST 匹配比字符串匹配更适合函数调用评估，因为函数调用的语义由函数名、参数名、参数值和嵌套结构决定，而不是由文本表面形式决定。例如 `get_weather(city="北京", date="today")` 和 `get_weather(date="today", city="北京")` 字符串不同，但调用语义相同。AST 可以忽略空格、参数顺序和部分格式差异，更准确地判断结构。

AST 也可能误判。假阳性例子：参数值类型不同但 AST 归一化后被当成相同，例如 `"7"` 和 `7` 在某些工具中含义不同；单位不同但数值相同，例如 `10` 表示人民币还是美元。假阴性例子：模型先调用 `convert_currency` 得到美元金额，再调用支付工具，最终等价于标准答案的一步调用，但 AST 与标准调用不同。改进方法包括：加入类型检查、参数 schema 校验、单位归一化、可交换参数规则、工具执行结果比较，以及对多步等价调用做执行级评估。

测试样本示例：

```text
simple:
1. 用户：“查询北京明天的天气。” 标准调用：get_weather(city="北京", date="tomorrow")
2. 用户：“把 250 美元换算成人民币。” 标准调用：convert_currency(amount=250, from="USD", to="CNY")
3. 用户：“给张三发一封主题为会议延期的邮件。” 标准调用：send_email(to="张三", subject="会议延期", body=...)

multiple:
1. 用户：“查北京天气，并帮我找附近的室内活动。” 标准调用：get_weather -> search_events
2. 用户：“查询订单状态，如果已发货再查物流。” 标准调用：get_order -> get_logistics
3. 用户：“查航班价格，再查目的地酒店。” 标准调用：search_flights -> search_hotels

parallel:
1. 用户：“同时查询北京、上海、广州今天空气质量。” 标准调用：三个 get_air_quality 可并行。
2. 用户：“比较 A、B、C 三只股票今天价格。” 标准调用：三个 get_stock_price 可并行。
3. 用户：“查三家供应商的库存。” 标准调用：三个 query_supplier_inventory 可并行。

irrelevance:
1. 用户：“告诉我什么是 MCP。” 可直接回答，不应调用天气、邮件等工具。
2. 用户：“请用一句话鼓励我。” 不应调用数据库。
3. 用户：“不要执行任何操作，只解释退款政策。” 不应调用 refund_order。
```

评估器扩展可以这样设计：

```python
class ToolCallEvaluation(BaseModel):
    function_correct: bool
    arguments_correct: bool
    order_correct: bool
    efficient: bool
    errors: list[str]

def evaluate_calls(predicted, expected, dependency_graph=None):
    normalized_pred = normalize_calls(predicted)
    normalized_exp = normalize_calls(expected)

    structure_score = compare_ast(normalized_pred, normalized_exp)
    order_score = check_order(normalized_pred, dependency_graph)
    efficiency_score = check_minimal_calls(normalized_pred, normalized_exp)
    error_report = classify_errors(normalized_pred, normalized_exp)

    return {
        "structure_score": structure_score,
        "order_score": order_score,
        "efficiency_score": efficiency_score,
        "errors": error_report,
    }
```

执行顺序评估需要依赖图：如果 B 使用 A 的结果，则 A 必须先执行；如果多个工具互不依赖，则顺序不应影响得分。效率评估要检查是否重复调用同一工具、是否调用了无关工具、是否用多个工具完成一个简单任务。错误报告应分类统计：函数选择错误、参数缺失、参数类型错误、顺序错误、多余调用、漏调用、越权调用。

## 习题 12.3：GAIA、自定义评估集与答案匹配

GAIA（General AI Assistants）评估智能体的综合能力。基于 12.3 节的内容，请完成以下扩展实践：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 在 12.3.2 节中介绍了 GAIA 的三个难度级别（Level 1/2/3）。请分析：这三个级别在任务复杂度、所需能力、评估标准等方面有什么差异？如果要设计 Level 4（超高难度），应该包含什么类型的任务？
- GAIA 使用“准精确匹配”算法来评估答案的正确性。请分析：这种方法如何处理答案的多样性（如“42”、“四十二”、“42.0”都应该被认为是正确的）？在什么情况下准精确匹配可能不够用？请设计一个更智能的答案匹配算法，能够处理语义等价的答案。
- 请基于 12.3.4 节的代码，实现一个“自定义 GAIA 评估集”：选择一个特定领域（如医疗、法律、金融），设计 10 个真实世界问题，并实现完整的评估流程。要求问题涵盖不同难度级别，并提供标准答案和评分标准。

### 参考答案

GAIA Level 1 通常是相对直接的任务，可能只需要一次检索、简单计算或读取文件即可得到答案。Level 2 需要多步操作，例如搜索多个来源、整合信息、做中等复杂推理。Level 3 需要更长链路的工具使用、跨来源验证、复杂文件处理和更强规划能力。难度越高，对智能体的计划、工具使用、纠错和信息整合能力要求越高。

如果设计 Level 4，应包含更接近真实工作的长期任务，例如需要跨多天数据、多个文件、多种工具、权限边界和中间检查点的任务。示例：分析一个企业季度经营异常，需要读取财务表、CRM 导出、客服工单、行业新闻，生成有证据引用的风险报告。Level 4 不应只难在信息多，还应难在任务分解、证据追踪、冲突处理和不确定性表达。

准精确匹配通常会做归一化：去空格、统一大小写、去标点、数值格式转换、同义数字转换。这样“42”“42.0”“四十二”可以归一为同一数值。它不够用的情况包括：开放解释题、答案包含多个条件、单位换算、同义短语、部分正确、需要排序或集合匹配的问题。

更智能的答案匹配算法可以分层：

```text
1. Exact Match：完全匹配。
2. Normalized Match：大小写、标点、空格、数字和单位归一化。
3. Set Match：适合多个实体答案，忽略顺序但检查集合。
4. Numeric Tolerance：允许小数误差和单位换算。
5. Semantic Match：用 NLI 或 LLM Judge 判断语义等价。
6. Evidence Check：要求答案引用的证据支持结论。
```

自定义金融 GAIA 评估集示例：

```text
Level 1:
1. 给定一张贷款还款表，计算某客户剩余本金。
2. 从公告中找出公司最近一次分红日期。
3. 根据汇率表换算 1000 美元对应人民币金额。

Level 2:
4. 对比两只基金近一年收益率和最大回撤，判断哪只风险更高。
5. 读取三份银行流水，统计某客户月均收入。
6. 根据贷款政策和客户资料判断是否需要人工复核。

Level 3:
7. 结合财报、新闻和股价数据，判断某公司利润下降的主要原因。
8. 从多份合同中识别可能影响授信的担保条款。
9. 分析客户交易记录，识别异常频繁转账模式。

Level 4:
10. 综合财报、行业数据、舆情新闻和现金流表，生成企业信用风险摘要，并列出证据来源和不确定性。
```

评估流程包括：准备输入文件和允许工具；为每题写标准答案、可接受答案变体、评分维度和证据位置；运行智能体；用分层匹配算法自动评分；对 Level 3/4 任务加入人工或 LLM Judge 复核；生成每题错误类型报告。

## 习题 12.4：LLM Judge 与多评委系统

LLM Judge 是使用大语言模型进行评估的新兴方法。基于 12.4 节的内容，请深入分析：

- 在 12.4.2 节中，我们使用 GPT-4 作为评判者来评估智能体的回答质量。请分析：LLM Judge 相比传统的规则匹配或指标计算有什么优势？它存在哪些潜在的偏见或局限性（如对某些回答风格的偏好、对长度的敏感性）？
- LLM Judge 的评分标准设计至关重要。请为以下三个不同的评估场景设计详细的评分标准（包括评分维度、权重、示例）：（1）代码生成质量评估；（2）创意写作质量评估；（3）技术文档质量评估。
- 在 12.4.3 节中提到，可以使用多个 LLM Judge 进行“评审团”式评估。请设计一个“多评委评估系统”：使用 3-5 个不同的 LLM（如 GPT-4、Claude、Qwen）作为评委，如何聚合它们的评分？如何处理评委之间的分歧？如何检测和过滤异常评分？

### 参考答案

LLM Judge 的优势是能评估开放式输出。规则匹配适合结构化答案，但难以判断一段解释是否完整、语气是否得体、代码设计是否清晰、文档是否易懂。LLM Judge 可以按 rubrics 对事实性、完整性、逻辑性、表达和安全性评分，也能给出具体原因。

局限包括：可能偏好更长、更流畅的回答；可能被回答中的自信语气影响；不同模型评分标准不一致；对自己家族模型输出可能有偏好；对事实正确性不一定可靠；评估成本较高；rubric 设计不清时会产生不稳定评分。因此高风险评估不应只依赖 LLM Judge，应结合规则、执行结果、人工抽检和证据校验。

代码生成质量评分标准：

```text
功能正确性 40%：是否满足需求，测试是否通过。
健壮性 20%：是否处理边界条件、异常输入和错误路径。
可读性 15%：命名、结构、注释是否清晰。
可维护性 10%：是否符合项目模式，是否避免不必要复杂度。
性能 10%：时间和空间复杂度是否合理。
安全性 5%：是否避免注入、敏感信息泄露、危险 API。
```

创意写作质量评分标准：

```text
主题契合 25%：是否符合题目和受众。
原创性 20%：是否有新意，避免模板化。
结构节奏 20%：开头、发展、转折、结尾是否自然。
语言表现 20%：词句是否生动、准确、有风格。
情感效果 10%：是否产生预期情绪。
约束遵守 5%：字数、体裁、视角等要求是否满足。
```

技术文档质量评分标准：

```text
准确性 30%：技术事实、命令、参数是否正确。
完整性 20%：是否覆盖安装、配置、使用、故障排查。
清晰度 20%：结构是否清楚，步骤是否可执行。
可验证性 15%：是否提供示例、输出和检查方法。
受众适配 10%：是否符合读者水平。
安全提示 5%：是否说明权限、数据和风险。
```

多评委系统可以让 3-5 个模型独立评分，所有评委使用同一 rubric、同一输入、同一输出格式。聚合方式可以是加权平均或中位数。中位数对异常值更稳健；加权平均可根据历史与人工评分的一致性给评委不同权重。

分歧处理流程：

```text
1. 收集所有评分和理由。
2. 计算均值、标准差和最大差值。
3. 如果分歧低于阈值，直接聚合。
4. 如果分歧较大，触发仲裁评委或人工复核。
5. 要求分歧评委引用 rubric 中具体条款，避免空泛判断。
```

异常评分检测包括：评分超出范围、理由与分数矛盾、未按格式输出、与其他评委偏差过大、长期评分分布异常、对同类样本明显过宽或过严。异常评分可以剔除或降低权重，但应记录原因，避免随意删除不喜欢的评审意见。

## 习题 12.5：分层评估、持续评估与报告生成

智能体评估的实践应用需要考虑多个方面。请思考：

- 在实际项目中，评估往往需要在“评估成本”和“评估质量”之间权衡。请设计一个“分层评估策略”：（1）快速评估（低成本，用于日常开发迭代）；（2）标准评估（中等成本，用于版本发布前）；（3）全面评估（高成本，用于重大更新或对外发布）。每层应该包含哪些评估项目？如何设计评估流程？
- 智能体的性能可能随时间变化（如依赖的外部 API 变化、用户需求变化）。请设计一个“持续评估系统”：能够定期自动运行评估，监控智能体性能的变化趋势，并在性能下降时及时告警。这个系统应该包含哪些组件？如何设计告警规则？
- 评估结果需要以清晰的方式呈现给不同的受众（如开发者、产品经理、用户）。请设计一个“评估报告生成系统”：能够根据受众类型自动生成不同详细程度的报告。开发者报告应该包含哪些技术细节？产品经理报告应该突出哪些业务指标？用户报告应该如何简化和可视化？

### 参考答案

分层评估策略可以按成本和发布风险划分。快速评估用于日常开发，每次改动后运行，包含小型核心用例集、工具调用格式检查、关键安全规则、少量回归测试和静态检查。它要求几分钟内完成，目标是快速发现明显回归。

标准评估用于版本发布前，包含更完整的业务场景集、工具调用正确率、意图分类准确率、异常处理测试、RAG 引用准确性、LLM Judge 抽评和性能指标。它可以运行几十分钟到数小时，目标是判断版本是否可发布。

全面评估用于重大更新或对外发布，包含大规模离线评估、人工评审、对抗测试、红队测试、线上灰度、A/B 测试、成本分析、延迟和稳定性压测、隐私合规检查。它成本最高，但能覆盖真实场景和高风险问题。

持续评估系统组件：

```text
Eval Dataset Store：保存评估集和版本。
Agent Runner：定期运行智能体。
Tool Sandbox：隔离执行外部工具。
Scorer：规则评分、执行评分、LLM Judge、人工评分入口。
Metrics Store：保存历史指标。
Dashboard：展示趋势。
Alert Manager：性能下降时告警。
Report Generator：生成评估报告。
```

告警规则可以包括：核心任务成功率较上个版本下降超过 3%；工具调用错误率超过 2%；安全违规样本数大于 0；P95 延迟超过阈值；成本上涨超过 20%；RAG 无引用回答比例上升；连续三次评估趋势下降。告警应区分阻断级、警告级和观察级，避免过度打扰。

报告生成系统应根据受众调整粒度。开发者报告需要技术细节：失败样本、输入输出、工具调用轨迹、错误分类、日志链接、版本 diff、延迟分布、成本、回归对比。产品经理报告应突出业务指标：任务成功率、用户满意度、转人工率、问题解决率、主要失败场景、上线风险和改进建议。用户报告应简化为可理解的结果：系统能力覆盖范围、可靠性水平、已知限制、最近改进和安全说明。

可视化设计可以包括：

```text
开发者：错误热力图、失败样本列表、工具调用链路。
产品经理：版本对比柱状图、业务指标趋势线、风险摘要。
用户：简单评分卡、能力清单、注意事项。
```

报告必须保留评估集版本、模型版本、运行时间和评分方法，否则不同版本之间不可比较。
