# 第十一章课后习题：Agentic-RL

> <strong>提示</strong>：部分习题没有标准答案，重点在于培养学习者对 Agentic RL 和智能体训练的综合理解和实践能力。

## 习题 11.1：从 LLM 训练到 Agentic RL

本章介绍了从 LLM 训练到 Agentic RL 的演进过程。请分析：

- 在 11.1.3 节的表 11.1 中，对比了 PBRFT（基于偏好的强化微调）和 Agentic RL 在 MDP 框架下的差异。请深入解释：为什么 Agentic RL 的状态空间 $s_t = (\text{prompt}, o_1, o_2, ..., o_t)$ 包含历史观察，而 PBRFT 的状态 $s_0 = \text{prompt}$ 只包含初始提示？这种差异对训练过程和最终效果有什么影响？
- 假设你要训练一个“智能代码调试助手”，它需要：（1）分析代码找出 bug；（2）查阅文档了解 API 用法；（3）修改代码；（4）运行测试验证修复效果。请将这个任务映射到强化学习框架，明确定义状态空间、行动空间、奖励函数和状态转移函数。
- 在 11.1.1 节中提到，传统监督学习存在“难以优化长期目标”的局限。请设计一个具体的多步推理任务（如数学证明、复杂问题求解），展示为什么监督学习难以优化中间步骤，而强化学习可以通过延迟奖励来解决这个问题。

### 参考答案

PBRFT 通常把一个 prompt 视为初始状态，模型生成一个完整回答，然后根据人类偏好或奖励模型优化回答质量。它主要优化“给定输入后输出什么答案更好”。在这个设置中，状态可以简化为 $s_0 = \text{prompt}$，因为模型没有真实地与外部环境进行多轮交互。

Agentic RL 面向的是多步智能体任务。智能体每一步会执行动作，例如搜索、调用工具、写代码、运行测试；环境会返回观察 $o_t$，例如搜索结果、工具返回值、错误日志、测试结果。下一步决策必须依赖历史观察，因此状态空间要包含 prompt 和到当前为止的观察序列：$s_t = (\text{prompt}, o_1, o_2, ..., o_t)$。如果不包含历史观察，智能体就无法知道哪些工具已经调用、哪些假设已被排除、测试为什么失败。

这种差异会影响训练过程和最终效果。PBRFT 更适合优化单轮回答偏好，例如更有帮助、更安全、更符合人类偏好。Agentic RL 则能优化长期任务成功率，例如“最终测试是否通过”“是否用最少工具完成任务”“是否避免危险操作”。训练难度也更高，因为奖励可能延迟到多步之后，动作空间更大，环境反馈不稳定，信用分配更困难。

智能代码调试助手可以映射为强化学习问题：

```text
状态空间 S：
  当前问题描述、代码片段、文件结构、历史工具调用、文档检索结果、已做修改、测试输出、错误日志。

行动空间 A：
  read_file(path)
  search_code(query)
  search_docs(api_name)
  edit_file(path, patch)
  run_tests(command)
  ask_clarification(question)
  finish(summary)

状态转移 T：
  执行动作后环境返回新观察。例如 read_file 返回文件内容，run_tests 返回测试日志，edit_file 改变代码库状态。

奖励函数 R：
  测试全部通过：+1.0
  修复原始 bug 且无回归：额外 +0.5
  引入新测试失败：-0.5
  无效工具调用：-0.1
  危险修改或越权操作：-1.0
  修改越小、步骤越少：小幅奖励
```

一个多步推理任务可以是几何证明。监督学习可以给模型大量“题目 -> 证明过程”的样本，但每一步证明是否必要、是否最优、是否能导向最终结论，很难只靠逐 token 模仿优化。模型可能学会表面格式，却在关键引理选择上失败。强化学习可以把“最终证明成立”作为延迟奖励，把中间步骤是否引入有效引理、是否违反已知条件、是否缩小目标差距作为部分奖励。这样即使某条证明路径与训练样本不同，只要最终正确，也能得到正向信号。

## 习题 11.2：SFT、LoRA 与 GRPO

SFT（监督微调）和 GRPO（群组相对策略优化）是本章的两个核心训练方法。基于 11.2 节和 11.3 节的内容，请深入思考：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 在 11.2.4 节的 SFT 训练代码中，我们使用了 LoRA（低秩适配）技术来减少训练参数。请分析：LoRA 的核心思想是什么？为什么它能够用少量参数（如 0.16%）实现接近全参数微调的效果？在什么情况下应该选择 LoRA 而不是全参数微调？
- GRPO 算法（11.3 节）相比传统的 PPO 算法有什么优势？请对比两者的训练流程，分析 GRPO 如何通过“群组相对奖励”来简化训练过程并提升稳定性。如果要将 GRPO 应用到其他任务（如代码生成、对话优化），需要做哪些调整？
- 请基于 11.2.5 节的代码，扩展 SFT 训练流程，添加以下功能：（1）支持多轮对话数据的训练；（2）添加数据增强策略（如同义改写、难度调整）；（3）实现训练过程的可视化监控（如 loss 曲线、样本质量评估）。

### 参考答案

LoRA 的核心思想是冻结原始大模型参数，只在部分线性层旁边加入低秩矩阵来学习参数更新。原本全参数微调需要更新大矩阵 $\Delta W$，LoRA 假设有效更新可以近似为低秩分解 $\Delta W = BA$，其中 $A$ 和 $B$ 的维度远小于原权重矩阵。训练时只更新这两个小矩阵，推理时可以把 LoRA 权重合并回原模型。

LoRA 能用少量参数取得接近全参数微调效果，是因为很多下游任务并不需要改变模型全部知识，只需要调整模型在特定任务上的行为分布。大模型本身已经具备语言、推理和世界知识，低秩适配足以改变输出风格、格式、领域术语和任务习惯。LoRA 适合显存有限、数据规模中小、需要快速适配多个任务、希望保留基础模型能力的场景。全参数微调更适合数据量很大、任务差异很大、需要深度改变模型能力且有足够算力和严格评估的场景。

PPO 通常需要策略模型、参考模型、奖励模型和价值模型，通过优势估计和裁剪目标稳定更新。流程较复杂，训练资源消耗较大。GRPO 使用同一 prompt 采样一组回答，根据组内相对奖励计算优势，弱化或省去单独价值模型的需求。它通过比较同组输出的好坏来估计相对优势，流程更简洁，显存和工程复杂度更低。

GRPO 的训练流程可以概括为：

```text
对同一问题采样多个回答
-> 用奖励函数分别打分
-> 计算组内均值和标准差
-> 将每个回答的奖励转成相对优势
-> 用相对优势更新策略，同时约束不要偏离参考模型太远
```

应用到代码生成时，奖励函数要从“最终答案是否正确”改为“测试是否通过、代码是否可运行、是否符合接口、是否安全”。应用到对话优化时，奖励函数要考虑帮助性、真实性、安全性、礼貌性和是否解决用户问题。不同任务都需要可靠的自动评估器，否则奖励噪声会直接影响训练。

扩展 SFT 支持多轮对话时，数据格式应保存 messages 列表：

```json
{
  "messages": [
    {"role": "system", "content": "你是代码助手"},
    {"role": "user", "content": "这个函数为什么报错？"},
    {"role": "assistant", "content": "请提供错误日志。"},
    {"role": "user", "content": "日志如下..."},
    {"role": "assistant", "content": "原因是..."}
  ]
}
```

训练时要使用聊天模板，把多轮消息拼接为模型输入，并只对 assistant 目标部分计算 loss。数据增强可以包括：同义改写用户问题、增加不同难度样本、打乱不影响语义的字段顺序、加入常见错误输入、生成负例并过滤低质量样本。监控方面，可以记录训练 loss、验证 loss、学习率、梯度范数、样本长度分布、格式错误率和小型评估集准确率，并用 TensorBoard、Weights & Biases 或本地可视化脚本展示曲线。

## 习题 11.3：奖励函数设计与奖励黑客

奖励函数设计是 Agentic RL 的核心挑战。基于 11.3.3 节的内容，请完成以下扩展实践：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 在 11.3.3 节中，我们为 GSM8K 数学问题设计了简单的二元奖励（正确+1，错误 0）。请设计一个更精细的奖励函数，能够：（1）对部分正确的答案给予部分奖励；（2）对推理过程的合理性进行评分；（3）惩罚过于冗长或低效的解题路径。这个奖励函数应该如何实现？
- 奖励函数的设计往往需要领域知识。请为以下三个不同的智能体任务设计奖励函数：（1）代码生成助手（需要考虑代码正确性、可读性、效率）；（2）客服对话智能体（需要考虑问题解决率、用户满意度、响应时间）；（3）游戏 AI（需要考虑胜率、策略多样性、对抗鲁棒性）。
- 在实际应用中，奖励函数可能存在“奖励黑客”（reward hacking）问题：智能体找到了获得高奖励的捷径，但并没有真正完成任务。请举例说明这种现象，并设计防御机制来避免奖励黑客。

### 参考答案

GSM8K 的精细奖励可以拆成答案正确性、过程合理性和效率三部分：

```text
reward =
0.50 * final_answer_score +
0.30 * reasoning_score +
0.10 * intermediate_step_score +
0.10 * efficiency_score
```

`final_answer_score` 判断最终数值是否正确，正确为 1，单位或格式轻微错误可给 0.8，数值接近但不完全正确可按误差给部分分。`reasoning_score` 判断解题步骤是否符合题意、是否有逻辑跳跃、公式是否合理。可以由规则和 LLM Judge 结合完成。`intermediate_step_score` 检查关键中间量是否正确，例如总价、人数、差值。`efficiency_score` 惩罚过长或重复推理，可以根据 token 数和步骤数计算；在正确答案前提下，越简洁越高分，但不能鼓励省略必要步骤。

代码生成助手奖励函数：

```text
0.45 * tests_pass_rate
0.20 * functional_correctness
0.15 * code_readability
0.10 * runtime_efficiency
0.05 * security_score
0.05 * minimal_change_score
```

其中测试通过率由真实测试给出，可读性可由 lint、复杂度和命名规则评估，安全分检查注入、硬编码密钥、危险 API 等。

客服对话智能体奖励函数：

```text
0.35 * issue_resolution
0.20 * policy_compliance
0.15 * user_satisfaction
0.10 * response_time_score
0.10 * factuality
0.10 * escalation_correctness
```

它既要解决问题，也要遵守公司政策和法律要求。对于低置信度或高风险问题，正确转人工应给正奖励。

游戏 AI 奖励函数：

```text
0.50 * win_rate
0.15 * long_term_advantage
0.10 * strategy_diversity
0.10 * resource_efficiency
0.10 * robustness_against_opponents
0.05 * rule_compliance
```

只奖励胜率可能导致策略单一或利用漏洞，因此要加入多样性、鲁棒性和规则合规。

奖励黑客例子：代码助手发现测试只覆盖了返回值，于是硬编码测试样例而不是实现通用逻辑；客服机器人为了提高响应速度，直接给出模板回复却没有解决问题；游戏 AI 利用环境漏洞卡住对手而不是学习真实策略。

防御机制包括：扩大和隐藏测试集，加入对抗样本；奖励函数多维度设计，避免单一指标；人工抽检高分样本；检测异常行为，例如过短回答、重复模板、硬编码；训练时随机化环境；对工具调用和最终结果做独立验证；上线前用真实场景回放评估。关键原则是奖励应尽量贴近真实目标，而不是只优化容易测量的代理指标。

## 习题 11.4：数学推理训练、泛化评估与在线学习

在 11.4 节的“数学推理智能体训练”案例中，我们看到了完整的训练流程。请深入分析：

- 案例中使用了 GSM8K 数据集进行训练和评估。请分析：这个数据集的特点是什么？它适合训练什么类型的推理能力？如果要训练一个能够处理更复杂数学问题（如高等数学、数学证明）的智能体，应该如何扩展数据集和训练方法？
- 在 11.4.3 节的训练结果中，我们观察到模型在训练集上的准确率提升，但可能存在过拟合风险。请设计一个“泛化能力评估”方案：如何测试模型是否真正学会了数学推理，而不是记住了训练数据？如何通过正则化、数据增强等技术提升泛化能力？
- 案例中的训练是离线的（使用预先收集的数据集）。请设计一个“在线学习”方案：智能体在实际使用过程中持续收集用户反馈，并自动更新模型。这个方案需要考虑哪些技术挑战（如数据质量控制、灾难性遗忘、安全性保障）？

### 参考答案

GSM8K 是小学到初中水平的英文数学应用题数据集，题目通常需要多步算术推理，答案有明确数值。它适合训练模型把自然语言问题转化为计算步骤，学习逐步推理、变量关系和基础算术。它不适合直接评估高等数学证明、抽象代数、实分析或复杂符号推导，因为这些任务需要更强的形式化推理、定理引用和长链证明能力。

如果要训练更复杂数学智能体，需要扩展数据集。高等数学可加入微积分、线性代数、概率论、优化、离散数学题目；数学证明可加入定理陈述、证明步骤、Lean/Coq 等形式化证明数据；还应加入错误证明识别、定理检索和工具使用样本。训练方法上，可以结合 SFT 学习规范推理格式，用 RL 优化最终正确性，用证明检查器或符号计算器提供可验证奖励。

泛化评估应避免只看训练集准确率。可以建立以下评估集：

```text
保留测试集：训练中完全不可见。
模板变体集：改变数字、实体和表述，检查是否理解结构。
难度外推集：步骤更多、干扰信息更多、数字范围更大。
跨数据集评估：使用其他数学题数据集。
对抗集：包含常见陷阱、单位转换、无关信息。
过程评估集：检查中间步骤是否合理。
```

如果模型在原题上高分，但在数字替换或表达改写后大幅下降，说明可能记忆了模式而非真正推理。提升泛化的方法包括：数据增强、题目重写、数字随机化、正则化、早停、验证集监控、混合多来源数据、限制过拟合训练轮数，以及在奖励函数中惩罚不合理推理。

在线学习方案可以分为“数据收集 -> 过滤 -> 标注 -> 训练 -> 评估 -> 灰度发布”。智能体在真实使用中收集用户反馈，例如答案是否有帮助、用户纠错、人工审核结果、工具验证结果。系统先过滤隐私数据和低质量样本，再把高价值样本加入训练池。训练时不应直接实时更新线上模型，而应周期性离线微调，经过评估后再灰度上线。

技术挑战包括：用户反馈噪声大，不能把所有点赞都当作正确；新数据可能分布偏移；持续训练可能导致灾难性遗忘；恶意用户可能投毒；隐私数据不能进入训练集；新模型可能引入安全回归。解决方案包括数据质量评分、人工审核高风险样本、保留回放数据集、混合旧数据训练、差分隐私或脱敏、模型版本管理、离线评估门禁、灰度发布和可回滚机制。

## 习题 11.5：工具学习、分层强化学习与课程学习

Agentic RL 的一个重要应用是让智能体学会使用工具。请思考：

- 在 11.1.3 节中提到，Agentic RL 适合优化“需要多步推理、工具使用、长期规划”的任务。请设计一个“工具学习”训练方案：给定一组工具（如搜索引擎、计算器、代码执行器），如何训练智能体学会在合适的时机选择合适的工具？奖励函数应该如何设计？
- 工具使用往往涉及复杂的依赖关系（如“必须先调用工具 A 获取信息，才能调用工具 B”）。请设计一个“分层强化学习”方案：高层策略负责任务规划，低层策略负责工具调用。这种分层结构如何训练？如何协调高层和低层的优化目标？
- 在实际应用中，工具的数量可能非常多（如 50+个 API），直接训练可能面临“探索效率低”的问题。请设计一个“课程学习”（curriculum learning）方案：从简单任务（使用少量工具）开始训练，逐步增加任务难度和工具数量。这个方案应该如何设计课程顺序？如何评估智能体是否准备好进入下一阶段？

### 参考答案

工具学习训练方案应提供一个可交互环境。环境包含任务、工具列表、工具 schema 和工具返回结果。智能体每一步可以选择回答、调用搜索、调用计算器、运行代码、读取文档等动作。训练样本应覆盖“必须用工具”“不需要工具”“工具会返回噪声”“需要多个工具组合”的场景。

奖励函数可以设计为：

```text
最终答案正确：+1.0
选择必要工具并得到有效结果：+0.2
不必要工具调用：-0.1
工具参数错误：-0.2
危险工具调用：-1.0
答案引用工具结果且事实一致：+0.2
步骤过多或成本过高：小幅惩罚
```

关键是不要奖励“多调用工具”，而要奖励“在合适时机调用合适工具”。例如简单常识题直接回答即可；精确计算题应调用计算器；最新信息题应搜索；代码行为题应运行测试或解释器。

分层强化学习可以把任务拆成高层规划和低层执行。高层策略输出子目标，例如“查找 API 文档”“计算结果”“验证代码”。低层策略根据子目标选择具体工具和参数。状态在两层共享，但抽象粒度不同。

训练方式可以先分别训练低层工具策略，让它学会在给定子目标下正确调用工具；再训练高层规划策略，让它学会生成合理子目标序列；最后端到端微调。高层奖励关注任务是否完成、规划是否简洁；低层奖励关注工具调用是否成功、参数是否正确、成本是否低。协调时，高层不能把不可执行目标交给低层，低层失败要把失败原因反馈给高层重新规划。

课程学习方案应从少工具、短链路、确定反馈开始，逐步增加复杂度：

```text
阶段 1：单工具任务，例如只用计算器或只用搜索。
阶段 2：工具选择任务，给 3-5 个工具，让模型判断是否需要工具。
阶段 3：双工具依赖任务，例如先搜索汇率再计算金额。
阶段 4：多工具任务，加入代码执行器、数据库、文档检索。
阶段 5：噪声和失败处理，工具可能超时、返回空结果或错误。
阶段 6：大工具集任务，50+ 工具，通过工具检索和路由缩小候选集。
```

进入下一阶段的标准应包括：当前阶段成功率达到阈值，工具参数错误率低于阈值，不必要调用率低于阈值，危险调用为零或极低，在保留集和改写任务上表现稳定。对于 50+ 工具场景，应引入工具目录、工具嵌入检索和候选工具路由，先让智能体从少量候选中选工具，而不是直接在全部工具中随机探索。
