# 第六章课后习题：框架开发实践

## 习题 6.1：对比智能体框架与协作哲学

本章介绍了四个各具特色的智能体框架：`AutoGen`、`AgentScope`、`CAMEL` 和 `LangGraph`。请分析：

- 在6.1.2节的表6.1中，对比了这四个框架的多个维度。请选择其中两个你最熟悉的框架，从“协作模式”、“控制方式”、“适用场景”三个维度进一步深入对比。
- 本章提到了“涌现式协作”与“显式控制”之间的权衡，如何理解这两种设计哲学的含义。

### 参考答案

可以选择 `AutoGen` 和 `LangGraph` 做对比。`AutoGen` 的协作模式偏对话式多智能体协作，多个角色通过群聊、轮询、选择发言者等方式共同推进任务。它更像一个“虚拟团队”，每个智能体有角色、人设和职责，任务结果来自多轮讨论、反馈和修正。`LangGraph` 的协作模式偏状态图和流程编排，开发者把任务拆成节点、边、状态和条件跳转，智能体只是图中的一个或多个节点。

在控制方式上，`AutoGen` 更强调智能体之间的自主交流，开发者通过系统消息、终止条件、群聊策略和工具权限影响协作，但对每一步路径的控制相对弱。`LangGraph` 则强调显式控制，开发者可以明确规定“先理解问题，再搜索，再回答；如果质量不够，则回到搜索或生成节点”。因此 LangGraph 的流程可预测性、可调试性和可审计性更强。

在适用场景上，`AutoGen` 适合需要多角色讨论、方案生成、代码协作、研究分析、创意写作等任务，因为这些任务允许一定程度的开放探索。`LangGraph` 适合对流程可靠性要求高、有明确分支和循环的生产系统，例如客服工单处理、风控审批、代码生成测试修复循环、RAG 问答流程等。

“涌现式协作”是指开发者不完全预设每一步，而是让多个智能体通过对话、角色分工和反馈自然形成解决方案。它的优点是灵活、创造性强，适合开放任务；缺点是结果不稳定，可能跑题、重复、循环或成本失控。

“显式控制”是指开发者把流程、状态、条件和终止规则写清楚，让智能体在受控框架内执行。它的优点是稳定、可测试、可审计、便于上线；缺点是前期设计成本更高，对未知问题的自适应能力较弱。工程上通常不是二选一，而是在关键流程上显式控制，在局部任务中允许智能体涌现式协作。

## 习题 6.2：扩展 AutoGen“软件开发团队”

在6.2节的 `AutoGen` 案例中，我们构建了一个“软件开发团队”。请基于此案例进行扩展思考：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 当前的团队使用 `RoundRobinGroupChat`（轮询群聊）模式，智能体按固定顺序发言。如果需求变更，工程师的代码需要返回给产品经理重新审核，应该如何修改协作流程？请设计一个支持“动态回退”的机制。
- 在案例中，我们通过 `System Message` 为每个智能体定义了角色和职责。请尝试为这个团队添加一个新角色“测试工程师”（`Quality Assurance`），并设计其系统消息，使其能够在代码审查后执行自动化测试。
- `AutoGen` 的对话式协作存在可能的不稳定性，可能导致对话偏离主题或陷入循环。请思考：如何设计一套“对话质量监控”机制，在检测到异常时及时干预？

### 参考答案

固定轮询适合演示，但不适合真实软件流程。支持动态回退时，可以把团队协作改造成“状态机 + 智能体群聊”的混合模式。状态至少包括：需求分析、产品审核、编码实现、代码审查、测试验证、修复、完成。每个智能体发言后输出结构化状态，例如 `status=needs_product_review`、`status=needs_fix`、`status=approved`。调度器根据状态选择下一个发言者，而不是固定轮询。

动态回退机制可以设计为：

```text
ProductManager 输出需求
-> Engineer 实现代码
-> Reviewer 审查
   -> 如果发现需求理解错误：回退 ProductManager 澄清需求
   -> 如果发现代码质量问题：回退 Engineer 修改
   -> 如果通过：进入 QA 测试
-> QA 执行测试
   -> 如果测试失败：回退 Engineer 修复
   -> 如果需求不一致：回退 ProductManager 重新确认
   -> 如果通过：结束
```

测试工程师的系统消息可以写成：

```text
你是软件开发团队中的 Quality Assurance 测试工程师。
你的职责是在代码审查通过后设计并执行测试。
你必须完成：
1. 根据产品需求提取可验证的测试点。
2. 编写或建议单元测试、集成测试和边界测试。
3. 如果可以执行命令，运行测试命令并记录结果。
4. 如果测试失败，必须给出失败用例、复现步骤、错误原因推断和修复建议。
5. 不要修改产品需求；如果发现需求不清晰，请标记为 NEEDS_PRODUCT_CLARIFICATION。
6. 输出必须包含：test_plan、test_cases、execution_result、risk_level、next_action。
只有当核心需求、边界条件和回归测试均通过时，才能输出 QA_APPROVED。
```

对话质量监控可以由一个独立的 Monitor 组件实现。它不参与业务讨论，只观察对话日志和结构化状态。监控指标包括：连续多轮没有新信息、同一问题重复出现、智能体输出偏离任务、工具调用连续失败、超过最大轮数、未遵守输出格式、成本超过预算、生成内容与需求不一致。检测到异常后可以采取三种干预：提醒当前智能体重新聚焦任务，强制切换到指定角色，或终止对话并要求人工介入。

工程实现上，应该为每轮对话记录 `speaker`、`message`、`state`、`action`、`tool_result`、`token_cost` 和 `quality_flags`。Monitor 可以用规则先做硬约束，例如超过 12 轮未完成就暂停；再用 LLM 做软判断，例如判断是否跑题。这样既能降低成本，又能提升稳定性。

## 习题 6.3：分析 AgentScope“三国狼人杀”

在6.3节的 `AgentScope` 案例中，我们实现了一个“三国狼人杀”游戏。请深入分析：

- 案例中使用了 `MsgHub`（消息中心）来管理智能体间的通信。请解释消息驱动架构相比传统函数调用的优势是什么？在什么场景下这种架构特别有价值？
- 游戏中使用了结构化输出（如 `DiscussionModelCN`、`WitchActionModelCN`）来约束智能体行为。请设计一个新的游戏角色“猎人”，并定义其对应的结构化输出模型，包括字段定义和验证规则。
- `AgentScope` 支持分布式部署，这意味着不同的智能体可以运行在不同的服务器上。请思考：在“三国狼人杀”这样的实时游戏场景中，分布式部署会带来哪些技术挑战？如何保证消息的顺序性和一致性？

### 参考答案

消息驱动架构的核心是智能体通过消息中心交换信息，而不是彼此直接调用函数。相比传统函数调用，它的优势是解耦。发送方只需要把消息发布到 `MsgHub`，不需要知道接收方的具体实现、位置和数量。接收方可以订阅相关消息并独立处理。这样更适合多智能体系统，因为智能体数量、角色、运行位置和消息范围都可能动态变化。

消息驱动架构还便于记录完整对话历史，支持广播、私聊、分组通信、异步处理和分布式部署。在狼人杀这类游戏中，白天讨论是广播，夜晚行动是私密消息，主持人需要控制消息可见范围，玩家需要根据历史发言推理身份，这些都非常适合用消息中心管理。在客服协作、机器人团队、仿真社会、多角色游戏和分布式任务执行中，消息驱动也很有价值。

“猎人”角色的结构化输出模型可以这样设计：

```python
from typing import Literal, Optional
from pydantic import BaseModel, Field, model_validator

class HunterActionModelCN(BaseModel):
    action: Literal["shoot", "skip"] = Field(description="猎人死亡后是否开枪")
    target_player_id: Optional[str] = Field(default=None, description="开枪带走的目标玩家 ID")
    reason: str = Field(min_length=5, max_length=200, description="行动理由")
    confidence: float = Field(ge=0.0, le=1.0, description="对目标判断的置信度")

    @model_validator(mode="after")
    def validate_target(self):
        if self.action == "shoot" and not self.target_player_id:
            raise ValueError("选择 shoot 时必须提供 target_player_id")
        if self.action == "skip" and self.target_player_id is not None:
            raise ValueError("选择 skip 时不能提供 target_player_id")
        return self
```

验证规则包括：`action` 只能是 `shoot` 或 `skip`；只有猎人死亡且规则允许开枪时才能调用该模型；选择开枪必须提供目标；目标必须是仍在游戏中的其他玩家；不能选择自己；理由不能为空；置信度必须在 0 到 1 之间。

分布式部署会带来延迟、消息乱序、重复投递、节点故障、状态不一致、时钟不同步和网络分区等问题。实时游戏对顺序性要求很高，例如夜晚行动必须按主持人规定阶段执行，投票必须在截止时间前统计，玩家不能看到不该看到的消息。

保证顺序性和一致性的方案包括：由中心化 GameMaster 或 MsgHub 维护权威状态；所有消息带有 `game_id`、`round_id`、`phase`、`sequence_number` 和时间戳；消息写入持久化日志后再广播；客户端按序列号处理消息；重复消息用 `message_id` 幂等去重；每个阶段使用状态机控制，只接受当前阶段合法动作；投票和夜晚动作在服务端统一结算；节点重连时从事件日志恢复状态。对强一致要求高的核心状态，应由单一权威服务或一致性存储维护，而不是让各智能体各自判断。

## 习题 6.4：CAMEL 协作与 Workforce

在6.4节的 `CAMEL` 案例中，我们让心理学家和作家协作创作电子书。

- 在案例中，协作会在检测到 `<CAMEL_TASK_DONE>` 标志时强制终止。但如果两个智能体意见分歧（一位认为可以终止，一位认为不应该终止），无法达成一致怎么办？请设计一个“冲突解决”的兼容机制。
- `CAMEL` 最初设计用于双智能体协作，但现在已经扩展支持多智能体。请查阅 `CAMEL` 的最新文档，了解其多智能体协作模块 [`workforce`](https://docs.camel-ai.org/key_modules/workforce)，并结合架构图说明其与 `AutoGen` 的群聊模式有何不同。

### 参考答案

只依赖 `<CAMEL_TASK_DONE>` 标志会过于脆弱，因为任一智能体都可能过早终止或拒绝终止。更稳妥的冲突解决机制应把终止判断从单个智能体输出中独立出来，增加任务完成度评估器。流程可以是：任一智能体提出完成，系统进入 `completion_review` 状态；两个智能体分别给出是否完成、理由和未完成项；独立评审器根据任务清单、质量标准和输出内容判断是否可以结束；如果分歧可修复，则生成补充任务；如果连续多轮仍分歧，则交给用户或更高优先级的仲裁智能体。

示例机制如下：

```text
Agent A 输出 <CAMEL_TASK_DONE>
-> 系统暂停普通对话
-> Agent A 和 Agent B 分别填写 completion_check
-> Evaluator 检查：
   1. 是否覆盖原始任务目标
   2. 是否存在明确未完成项
   3. 输出是否达到最低质量标准
   4. 是否超过最大轮数或预算
-> 结果：
   - complete：结束
   - revise：生成补充任务并继续
   - escalate：请求人工确认
```

`CAMEL Workforce` 是 CAMEL 的多智能体协作引擎，用于组装、管理和扩展由多个专门智能体组成的团队。它包含 `Workforce` 编排器、任务分解智能体、协调智能体、不同类型的 worker、可选共享记忆、任务超时、回调和结构化输出处理。其重点是把复杂任务分解给专业 worker，并通过协调器管理生命周期。

架构可以表示为：

```text
User Task
-> Workforce Orchestrator
   -> Task Agent：拆解任务
   -> Coordinator Agent：分派任务、收集结果
   -> Worker A：检索/研究
   -> Worker B：写作/生成
   -> Worker C：审查/修订
   -> Shared Memory（可选）
-> Final Result
```

`AutoGen` 群聊模式更像会议室：多个智能体在同一个对话空间里轮流或按策略发言，结果来自对话互动。`CAMEL Workforce` 更像项目管理系统：任务被分解、分派给 worker，协调器负责组织执行和汇总。AutoGen 的优势是对话自然、适合开放讨论；Workforce 的优势是结构更清晰，更适合多角色并行处理复杂任务。两者都能做多智能体协作，但控制中心和任务组织方式不同。

## 习题 6.5：LangGraph 状态图与循环机制

在6.5节的 `LangGraph` 案例中，我们构建了一个“三步问答助手”。请分析：

- `LangGraph` 将智能体流程建模为状态机和有向图。请画出案例中“理解-搜索-回答”流程的图结构，标注节点、边和状态转换条件。
- 当前的助手是一个线性流程。请扩展这个案例，添加一个“反思”节点：如果生成的答案质量低（例如过于简短或缺乏细节），系统应该重新搜索或重新生成答案。请设计这个循环机制的条件边逻辑。
- `LangGraph` 的优势在于对循环的原生支持。请设计一个更复杂的应用场景，充分利用这一特性：例如“代码生成-测试-修复”循环、“论文写作-审阅-修改”循环等。要求画出完整的图结构并说明关键节点的功能。

### 参考答案

“理解-搜索-回答”的基础图结构可以表示为：

```text
START
-> understand_question
-> search_information
-> generate_answer
-> END
```

状态对象可以包含：

```text
question：用户原始问题
intent：问题意图
search_query：检索查询
documents：检索结果
answer：最终答案
```

边和转换条件如下：`START -> understand_question` 无条件进入；`understand_question -> search_information` 在成功生成检索词后进入；`search_information -> generate_answer` 在获得检索结果后进入；`generate_answer -> END` 在答案生成完成后结束。如果理解失败，可以增加 `ask_clarification` 节点；如果搜索为空，可以转到 `rewrite_query` 节点。

加入反思节点后，图结构可以改为：

```text
START
-> understand_question
-> search_information
-> generate_answer
-> reflect_answer
   -> END：答案质量合格
   -> search_information：缺少事实或证据
   -> generate_answer：事实足够但表达质量差
   -> ask_clarification：问题本身不清晰
```

条件边逻辑可以基于结构化评分：

```text
reflect_answer 输出：
score: 0-10
has_enough_evidence: true/false
is_too_short: true/false
needs_clarification: true/false
next_action: finish | research_more | regenerate | clarify
```

如果 `score >= 8` 且证据充分，则结束；如果 `has_enough_evidence=false`，回到搜索节点，并把缺失信息转化为新的查询；如果 `is_too_short=true` 或结构混乱，则回到生成节点；如果用户问题不明确，则进入澄清节点。为避免无限循环，应设置最大循环次数，例如最多反思 2 次。

更复杂的“代码生成-测试-修复”循环可以设计为：

```text
START
-> analyze_requirement：理解需求和接口约束
-> generate_code：生成代码
-> run_tests：运行单元测试/类型检查/lint
-> evaluate_result
   -> END：测试全部通过
   -> diagnose_failure：测试失败
   -> ask_user：需求不清晰或缺少信息
-> diagnose_failure：分析错误日志和失败用例
-> patch_code：修复代码
-> run_tests
```

关键节点功能如下：`analyze_requirement` 提取输入输出、边界条件和验收标准；`generate_code` 只负责实现；`run_tests` 调用真实测试工具；`evaluate_result` 根据退出码和日志决定下一步；`diagnose_failure` 解释失败原因，避免盲目修改；`patch_code` 做最小修复。这个场景充分利用了 LangGraph 的循环和条件边能力，把“不通过就修复再测”的工程过程建模成可控状态机。

## 习题 6.6：智能体框架选型

框架选型是智能体产品开发过程中的关键决策之一。假设你是一家 `AI` 公司的技术架构师，公司计划开发以下三个智能体产品应用，请为每个应用选择最合适的框架（`AutoGen`、`AgentScope`、`CAMEL`、`LangGraph` 或不借助框架从零开发），并详细说明理由：

<strong>应用A</strong>：智能客服系统，需要处理大量并发用户请求（每秒1000+），要求响应时间低于2秒，系统需要7×24小时稳定运行，并支持水平扩展。

<strong>应用B</strong>：科研论文辅助写作平台，需要一个“研究员智能体”和一个“写作智能体”深度协作，共同完成文献综述、实验设计、数据分析和论文撰写。要求智能体能够进行多轮深度讨论，自主推进任务。

<strong>应用C</strong>：金融风控审批系统，需要按照严格的流程处理贷款申请：资料审核 → 风险评估 → 额度计算 → 合规检查 → 人工复核 → 最终决策。每个环节都有明确的判断标准和分支逻辑，要求流程可追溯、可审计。

### 参考答案

应用 A 不宜直接依赖开放式多智能体群聊框架作为核心链路，更适合从零开发高性能服务框架，并在局部使用轻量智能体能力。原因是每秒 1000+ 请求、低于 2 秒响应、7×24 小时稳定运行和水平扩展，首先是后端工程和系统架构问题。核心应包括 API 网关、负载均衡、缓存、队列、模型网关、熔断限流、FAQ/RAG 检索、人工转接和监控告警。智能体框架可以用于复杂工单分析，但不能让多轮对话式协作阻塞主链路。

如果必须在给定框架中选择，`LangGraph` 相对更合适，因为客服流程可以建模为可控图：识别意图、检索知识、调用订单工具、生成回复、置信度判断、转人工。它比 AutoGen 式群聊更稳定、可审计、易设置超时和分支。但在超高并发下，仍需要把 LangGraph 作为业务编排层嵌入到自研服务中，而不是直接替代服务架构。

应用 B 适合选择 `CAMEL` 或 `AutoGen`。题目强调研究员智能体和写作智能体深度协作、多轮讨论、自主推进任务，这正是对话式或角色协作式框架的优势。若任务主要是两个角色反复讨论，可以使用 CAMEL 的角色扮演机制；如果后续要加入审稿人、实验分析师、引用检查员等多个角色，可以使用 CAMEL Workforce 或 AutoGen 群聊。

选择 CAMEL 的理由是它天然强调角色扮演和任务协作，适合“研究员提出观点，写作智能体组织文本，再由研究员补充证据”的循环。它能让两个角色保持明确分工。需要注意的是，论文写作涉及事实和引用，必须接入文献检索工具、引用管理、事实核查和人工确认，不能让模型编造论文来源。

应用 C 最适合选择 `LangGraph`。金融风控审批有严格流程、明确分支、强审计要求和人工复核节点，不能依赖涌现式对话自然推进。LangGraph 可以把每个审批环节建模为节点，把判断标准建模为条件边，把申请状态、证据、评分、审批意见和人工复核记录保存在状态对象中。

可设计为：

```text
资料审核
-> 资料不全：补充资料
-> 资料完整：风险评估
-> 额度计算
-> 合规检查
-> 高风险或高金额：人工复核
-> 最终决策
-> 审计归档
```

这样每一步都有输入、输出、规则、日志和责任边界。对于金融系统，还应加入规则引擎、模型评分服务、权限控制、可解释性报告、审批日志、防篡改存储和人工兜底。AutoGen 或 CAMEL 可用于生成分析说明，但不应控制核心审批流程。
