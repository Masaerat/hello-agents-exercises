# 第一章参考答案

> 说明：以下答案是参考答案，不是唯一标准答案。优秀答案应准确使用第一章概念，并能用具体理由支撑判断。

## 参考答案 1.1：判断智能体类型

| 案例 | 是否属于智能体 | 可能类型 | 理由 |
| --- | --- | --- | --- |
| Case A：超级计算机 | 不一定属于 | 单独看不是智能体 | 算力强不等于具备智能体属性。智能体需要能够感知环境、基于目标自主决策，并通过行动影响环境。超级计算机可以运行智能体程序，但硬件本身只是计算平台。 |
| Case B：自动驾驶系统 | 属于 | 反应式智能体、基于模型的智能体、混合式智能体 | 自动驾驶系统通过摄像头、雷达、激光雷达等传感器感知道路环境，并通过刹车、转向、加速等执行器采取行动。障碍物避让强调毫秒级响应，因此具有强反应式特征；同时它也会维护车辆、车道和障碍物的内部状态模型，因此通常不是简单反射智能体，而是混合式系统。 |
| Case C：AlphaGo | 属于 | 基于目标、基于效用、学习型、规划式智能体 | AlphaGo 观察棋盘状态，评估候选落子，并选择行动。它的目标是赢棋，因此是目标导向的；它会评估局面价值和胜率，因此具有效用评估；它通过数据和自我博弈学习策略，因此也是学习型智能体；它会向未来多步展开搜索，因此属于规划式智能体。 |
| Case D：智能客服 | 属于，但前提是接入工具和业务环境 | LLM 驱动智能体、目标导向智能体、混合式智能体 | 如果 ChatGPT 只是被动聊天，它更像对话模型；但题目中它能够查询订单、分析投诉、提出方案并安抚用户，说明它能够通过用户消息和业务 API 感知环境，通过回复和工具调用采取行动，并围绕“解决投诉”这一目标进行推理，因此可视为 LLM 驱动的智能体。 |

**评分要点：**

- 不应把“高算力”直接等同于“智能体”。
- 判断时应关注环境、传感器、执行器、目标和自主性。
- 一个案例可以同时属于多个分类维度，不必强行只选一类。

## 参考答案 1.2：为智能健身教练设计 PEAS 模型

| PEAS 项 | 参考描述 |
| --- | --- |
| Performance 性能度量 | 训练安全性、目标达成度、运动效果、计划个性化程度、用户坚持率、伤病预防、用户满意度、长期健康改善、隐私保护。 |
| Environment 环境 | 用户身体状态、家庭或健身房空间、运动器械、可穿戴设备、用户时间安排、饮食习惯、健康史、室外运动时的天气、可能参与的教练或医生。 |
| Actuators 执行器 | 语音指导、App 提醒、训练计划调整、计时器、动作纠正提示、饮食建议、风险告警、暂停训练建议、与健身设备联动。 |
| Sensors 传感器 | 心率、运动强度、加速度计、陀螺仪、摄像头或姿态识别数据、用户反馈、训练日志、睡眠数据、体脂体重数据、饮食记录、用户输入的健康限制。 |

**环境特性分析：**

- 部分可观察：智能体无法直接知道用户的全部身体状态、疲劳程度、疼痛感、真实动机或完整动作质量，只能通过传感器和用户反馈间接判断。
- 随机性：同样的训练计划对不同用户、不同睡眠和饮食状态下的效果不同，用户也可能临时中断或动作变形。
- 序贯性：今天的训练会影响明天的疲劳、恢复、伤病风险和后续计划。
- 动态性：用户身体状态在运动中持续变化，智能体需要实时调整强度和提示。
- 连续性：心率、速度、关节角度、运动强度等信号通常是连续数值。
- 多智能体性：在部分场景下，用户、真人教练、医生、其他健康 App 或设备算法都会影响环境。

**评分要点：**

- PEAS 不能只写抽象词，需要和健身场景对应。
- 性能度量必须包含安全性，不能只追求目标达成。
- 环境特性分析要给出理由。

## 参考答案 1.3：电商退款审核中的 Workflow 与 Agent

**Workflow 的优点：**

- 规则明确，行为稳定，可预测。
- 容易审计和解释，适合合规要求较高的流程。
- 对高频、重复、边界清晰的申请处理效率高。
- 实现和维护成本相对可控。

**Workflow 的缺点：**

- 面对复杂或模糊情况时不够灵活。
- 规则变多后容易变得臃肿，维护成本上升。
- 难以理解自然语言投诉、图片证据、上下文情绪等非结构化信息。
- 对合理但不符合固定规则的特殊案例，可能造成较差用户体验。

**Agent 的优点：**

- 能理解自然语言投诉，并综合多种上下文信息。
- 适合处理规则边界模糊、证据复杂、需要解释和沟通的场景。
- 可以主动追问缺失信息，生成更个性化的处理方案。
- 当工具和知识库更新及时，它能适应更复杂的业务变化。

**Agent 的缺点：**

- 可能产生幻觉或不一致决策。
- 如果缺少日志和约束，审计难度更高。
- 可能错误理解政策或越权处理高风险退款。
- 需要完善评估、监控、权限控制和人工兜底。

**Workflow 更适合的情况：**

- 政策稳定、边界清晰、判断条件结构化。
- 风险较低，或风险可以通过金额、品类等规则清楚界定。
- 公司更重视一致性、合规性和可审计性。
- 大多数申请都是标准化、重复性请求。

**Agent 更有优势的情况：**

- 用户描述复杂，包含大量自然语言或非结构化证据。
- 需要结合历史行为、商品状态、客服记录、风控信号综合判断。
- 需要高质量沟通、解释、安抚和追问。
- 业务希望系统能在复杂边界情况中给出建议。

**推荐的方案 C：混合方案**

更稳妥的做法是“规则兜底 + 智能体处理复杂案例”：

- 对明显符合规则的低风险申请，使用 Workflow 自动通过或拒绝。
- 对高金额、特殊商品、疑似欺诈、证据复杂的申请，交给 Agent 汇总信息并给出建议。
- 对高风险或敏感案例保留人工审批。
- Agent 的工具调用、证据来源、判断过程和最终建议都要记录，便于审计。
- 设定硬性规则，例如法律要求、黑名单、金额上限等，Agent 不能越权绕过。

**评分要点：**

- 不应简单认为 Agent 一定优于 Workflow。
- 能提出混合方案，并说明边界和兜底机制，是较高质量答案。

## 参考答案 1.4：扩展智能旅行助手

**1. 添加记忆功能**

可以把记忆分为两类：

- 用户长期偏好：喜欢历史文化景点、预算范围、是否偏好室内活动、是否愿意长距离步行、饮食偏好等。
- 当前会话状态：本轮已推荐景点、被拒绝的选项、拒绝原因、当前城市、天气、门票状态等。

循环可以调整为：

```text
Observation -> 更新会话状态/记忆 -> Thought -> 选择工具 -> Action -> Observation
```

系统提示词中应要求智能体在推荐前读取用户偏好，并在用户反馈后更新记忆。长期稳定偏好可以写入长期记忆；临时信息如“今天下雨”“本轮推荐过颐和园”更适合保留在会话状态中。

**2. 处理门票售罄**

需要新增工具，例如：

```text
check_ticket(attraction, date) -> available | sold_out | unknown
search_alternative_attractions(city, constraints) -> list
```

当 `check_ticket` 返回 `sold_out` 时，智能体不应直接结束，而应把售罄结果作为新的 Observation，重新规划：

```text
Thought: 当前景点门票已售罄，需要基于用户偏好、天气和预算搜索替代方案。
Action: search_alternative_attractions(city="北京", constraints="历史文化, 室外, 预算适中")
```

**3. 连续拒绝后的反思**

需要记录拒绝次数和原因：

```text
state.rejected_recommendations = [
  { attraction: "A", reason: "太贵" },
  { attraction: "B", reason: "太远" },
  { attraction: "C", reason: "不感兴趣" }
]
```

当连续拒绝达到 3 次时，智能体应进入反思或澄清步骤：

```text
Thought: 用户连续拒绝了 3 个推荐，说明我可能误解了偏好或约束。应先澄清关键偏好，而不是继续盲目推荐。
Action: ask_user(question="你更希望我优先考虑价格、距离、室内活动，还是人少安静的景点？")
```

**停止条件和安全边界：**

- 保留最大循环次数，避免无限重试。
- 当用户接受推荐、缺少关键信息需要追问、没有可靠候选项、或达到最大循环次数时停止。
- 不应编造门票状态；如果工具返回 unknown，应明确说明不确定并建议用户确认。

**评分要点：**

- 必须体现 `Thought-Action-Observation` 循环的变化。
- 记忆必须参与后续决策，而不是只保存文本。
- 门票售罄应通过观察结果触发重新规划。

## 参考答案 1.5：智能体中的系统 1 与系统 2

以下以“金融风控智能体”为例。

**适合系统 1 处理的任务：**

- 快速识别常见欺诈模式。
- 对明显低风险或高风险交易进行初步打分。
- 识别异常设备、异常登录地点、异常金额、异常交易频率。
- 在毫秒级给出风险预警。

这些任务适合系统 1，因为它们强调快速模式识别和实时响应。

**适合系统 2 处理的任务：**

- 分析多个风险信号相互矛盾的复杂案例。
- 检查明确的合规规则和业务政策。
- 解释为什么拦截、放行或要求二次验证。
- 在放行、延迟、拦截、人工审核之间权衡。
- 对用户申诉或事后审计进行推理。

这些任务适合系统 2，因为它们需要慢速推理、规则检查、可解释性和责任边界。

**协同方式：**

1. 系统 1 先对交易进行快速风险评分，并标出可疑特征。
2. 系统 2 对中高风险或高影响交易进行规则核查和上下文推理。
3. 智能体根据综合判断选择放行、二次验证、拦截或转人工。
4. 事后根据真实欺诈、误杀和用户申诉结果，更新模型和规则。

**其他合理场景：**

- 医疗诊断助手：系统 1 识别影像或症状模式，系统 2 做鉴别诊断、检查禁忌和治疗建议。
- 法律咨询机器人：系统 1 检索相似案例，系统 2 根据法律条文进行结构化推理。
- 智能客服：系统 1 识别意图和情绪，系统 2 根据政策制定解决方案。

**评分要点：**

- 不应把系统 1 简单理解为“低级”，也不应把系统 2 简单理解为“高级”。两者解决的问题不同。
- 高质量答案需要说明两者如何交接、校验和反馈。

## 参考答案 1.6：LLM 智能体的局限与评估

**为什么会产生幻觉：**

- 大语言模型本质上是在生成概率上合理的文本，而不是直接保证事实正确。
- 模型可能缺少最新、私有或专业领域信息。
- 用户提示不明确时，模型可能用看似合理的假设补全缺失信息。
- 工具返回的信息可能缺失、过期、噪声较多，或被智能体错误解释。
- 多轮循环中，小错误可能在后续步骤不断累积。
- 如果没有外部验证，智能体可能过度相信自己的中间推理。

**如果取消最大循环次数，可能出现的问题：**

- 无限循环：反复调用同一个工具或重复失败计划。
- 成本失控：token、API 调用、外部服务费用持续增加。
- 重复副作用：重复下单、重复发消息、重复写数据库等。
- 目标漂移：执行过程中逐渐偏离用户最初目标。
- 错误放大：错误观察或错误假设影响后续多轮决策。
- 用户体验变差：用户长时间等待，却得不到明确结果。

**缓解思路：**

- 设置最大循环次数、超时时间和预算上限。
- 为任务设计明确停止条件。
- 对事实性结论要求工具验证或来源支撑。
- 使用结构化输出，并校验 Action 格式。
- 记录 Thought、Action、Observation 轨迹，便于调试和审计。
- 对不确定、高风险或多次失败的任务转人工处理。

**如何评估智能体的智能程度：**

仅使用准确率不够。智能体会与环境交互、使用工具、进行多步规划，因此评价维度应更全面：

- 任务成功率：是否真正完成用户目标。
- 事实可靠性：关键事实是否有来源或工具结果支撑。
- 鲁棒性：面对缺失信息、异常输入、工具失败时能否稳定处理。
- 效率：完成任务需要多少轮循环、token 和工具调用。
- 安全性：是否避免越权、危险、不可逆或违反政策的行动。
- 可解释性：用户或审核者能否理解其决策依据。
- 适应性：能否根据观察结果和用户反馈调整计划。
- 用户满意度：沟通是否清晰，方案是否解决真实需求。
- 长程可靠性：多步骤任务中是否能保持目标一致和稳定表现。

**评分要点：**

- 幻觉分析应联系 LLM 的生成机制和缺少外部 grounding 的问题。
- 最大循环次数应被理解为工程安全机制，而不是随意限制。
- 评价指标应覆盖任务完成、可靠性、安全性、效率和用户体验。
