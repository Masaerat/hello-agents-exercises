## 习题 8.1：四种记忆类型与健康管理助手

本章介绍了四种记忆类型：工作记忆、情景记忆、语义记忆和感知记忆。请分析：

- 在8.2.5节中，每种记忆类型都有独特的评分公式。请对比情景记忆和语义记忆的评分机制，解释为什么情景记忆更强调“时间近因性”（权重0.2），而语义记忆更强调“图检索”（权重0.3）？
- 如果要设计一个“个人健康管理助手”（需要记录用户的饮食、运动、睡眠数据，并提供健康建议），你会如何组合使用这四种记忆类型？请为每种记忆类型设计具体的应用场景。
- 工作记忆采用TTL（Time To Live）机制自动清理过期数据。请思考：在什么情况下，重要的工作记忆应该被“整合”（consolidate）为长期记忆？如何设计一个自动整合的触发条件？

### 参考答案

情景记忆记录的是“某个时间、某个场景下发生了什么”，例如用户昨天晚饭吃了什么、上周跑步时膝盖疼、某次对话中表达了睡眠焦虑。这类记忆的价值通常和时间强相关，越近的事件越可能影响当前判断，所以评分公式更强调时间近因性。对于健康助手来说，今天的低血糖、最近三天的失眠、刚刚完成的运动，比两年前一次普通晚餐更重要。

语义记忆记录的是更稳定的知识、概念和关系，例如“用户对花生过敏”“有氧运动有助于改善心肺功能”“高盐饮食与血压升高相关”。这类信息不一定因为时间变远就失去价值，更重要的是能否通过实体和关系推理找到相关知识。因此语义记忆更强调图检索。图检索可以发现多跳关系，例如“用户父亲有糖尿病 -> 用户关注控糖 -> 高 GI 食物 -> 早餐建议避免甜饮料”，这不是简单时间排序能解决的。

个人健康管理助手可以这样组合四种记忆：

```text
工作记忆：保存当前对话和短期任务状态，例如“用户正在记录今天午餐”“本轮还缺少运动时长”。
情景记忆：保存具体健康事件，例如“2026-07-06 晚上睡眠 5 小时”“周三跑步 4 公里后膝盖疼”。
语义记忆：保存稳定偏好、医学常识和关系，例如“用户乳糖不耐受”“用户目标是减脂”“高血压人群应控制钠摄入”。
感知记忆：保存来自图片、设备和传感器的信息，例如餐盘照片识别出的食物、手环心率、睡眠阶段、体重秤数据。
```

工作记忆应在短期内服务当前任务，但当它满足“长期有用”的条件时应整合为长期记忆。触发条件可以包括：用户明确表达偏好或限制，例如“我不能喝牛奶”；同一信息在多轮对话中重复出现，例如连续三次提到睡眠不足；信息被标记为高重要性或高风险，例如过敏、慢性病、用药；短期数据形成稳定趋势，例如连续两周平均睡眠低于 6 小时。自动整合可以采用评分：

```text
consolidation_score =
0.35 * importance +
0.25 * repeat_count_score +
0.20 * health_risk_score +
0.10 * user_confirmation +
0.10 * recency
```

当分数超过阈值时，系统把工作记忆整理为情景记忆或语义记忆；对敏感健康信息，最好先让用户确认再长期保存。

## 习题 8.2：RAG 分块、检索策略与嵌入模型选型

在8.3节的RAG系统中，我们使用MarkItDown将各种格式文档统一转换为Markdown。请深入思考：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 当前的智能分块策略基于Markdown的标题层次（#、##、###）进行分割。如果处理的是没有明确标题结构的文档（如小说、法律条文），应该如何优化分块策略？请尝试实现一个基于“语义边界”的分块算法。
- 在8.3.5节中介绍了MQE（多查询扩展）和HyDE（假设文档嵌入）两种高级检索策略。请选择一个实际场景（如技术文档问答、医疗知识检索），对比基础检索、MQE和HyDE三种方法的效果差异，并分析各自的适用场景。
- RAG系统的检索质量很大程度上取决于嵌入模型的选择。请对比本章提到的三种嵌入方案（百炼API、本地Transformer、TF-IDF），从准确性、速度、成本、离线部署等维度进行评估，并给出选型建议。

### 参考答案

没有明确标题结构的文档不能只靠 Markdown 标题分块。小说可以按场景、段落主题、人物对话密度和时间地点变化分块；法律条文可以按条、款、项、但书、定义和引用关系分块。语义边界分块的目标是让每个块内部主题一致、块之间边界自然，同时控制长度。

一个基于语义边界的算法如下：

```python
def semantic_chunk(paragraphs, embed, max_tokens=800, min_tokens=200, threshold=0.72):
    chunks = []
    current = []
    current_tokens = 0
    last_embedding = None

    for para in paragraphs:
        para_tokens = count_tokens(para)
        para_embedding = embed(para)

        should_split = False
        if current and last_embedding is not None:
            similarity = cosine_similarity(last_embedding, para_embedding)
            if similarity < threshold and current_tokens >= min_tokens:
                should_split = True
            if current_tokens + para_tokens > max_tokens:
                should_split = True

        if should_split:
            chunks.append("\n".join(current))
            current = [para]
            current_tokens = para_tokens
        else:
            current.append(para)
            current_tokens += para_tokens

        last_embedding = para_embedding

    if current:
        chunks.append("\n".join(current))

    return chunks
```

法律条文还应增加规则边界：遇到“第 X 条”“一、”“（一）”“但是”“除非”等结构时优先作为分块候选；小说可在时间地点变化、人物视角变化、空行和章节符号处提高切分概率。

以技术文档问答为例，基础检索直接用用户问题向量召回文档，速度快、成本低，适合问题表述和文档表述接近的场景，例如“如何配置 API Key”。MQE 会把用户问题改写成多个查询，例如“API Key 配置”“环境变量设置”“认证失败”，再合并结果，适合用户问题模糊、同义表达多的场景。HyDE 会先让模型生成一个假设答案或假设文档，再用该文本做向量检索，适合用户问题很短、缺少关键词或需要概念扩展的场景。

三者效果差异通常是：基础检索最快最便宜，但召回可能不足；MQE 召回更全面，但会增加多次检索和去重成本；HyDE 对抽象问题有帮助，但如果假设文档偏离事实，可能引入错误方向。生产系统可以先用基础检索，召回不足时启用 MQE；对开放性问题或概念性问题启用 HyDE；最终用重排序模型和引用校验控制质量。

三种嵌入方案对比如下：

```text
百炼 API：准确性通常较好，接入简单，适合中文和通用语义检索；缺点是需要联网、按量计费、数据会经过外部服务。
本地 Transformer：准确性取决于模型，使用 bge、gte 等模型可获得较好效果；可离线部署，数据安全更好；缺点是需要显存/CPU 资源和模型维护。
TF-IDF：速度快、成本低、可完全离线，适合关键词匹配、术语精确匹配和小规模文档；缺点是不理解语义，同义词和改写召回较弱。
```

选型建议：公开数据、快速原型可以用百炼 API；企业私有知识库、敏感数据和离线环境优先本地 Transformer；需要关键词精确匹配、字段检索或资源极低的场景可以用 TF-IDF。实际系统中常用混合检索：向量检索负责语义召回，BM25/TF-IDF 负责关键词召回，再用重排序模型融合。

## 习题 8.3：遗忘、归档与敏感信息删除

记忆系统的“遗忘”机制是模拟人类认知的重要设计。基于8.2.3节的MemoryTool，请完成以下扩展实践：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 当前提供了三种遗忘策略：基于重要性、基于时间、基于容量。请设计并实现一个“智能遗忘”策略，综合考虑重要性、访问频率、时间衰减等多个因素，使用加权评分来决定哪些记忆应该被遗忘。
- 在长期运行的智能体系统中，记忆数据库可能会积累大量数据。请设计一个“记忆归档”机制：将长期不用但可能有价值的记忆转移到冷存储，需要时再恢复。这个机制应该如何与现有的四种记忆类型集成？
- 思考：如果智能体需要“忘记”某些敏感信息（如用户隐私数据），仅仅从数据库删除是否足够？在使用向量数据库和图数据库的情况下，如何确保数据被彻底清除？

### 参考答案

智能遗忘策略可以给每条记忆计算保留分，分数越低越应该被遗忘。评分要同时考虑重要性、访问频率、最近访问时间、创建时间、用户标记和风险级别。示例公式：

```text
retention_score =
0.35 * importance +
0.20 * access_frequency +
0.20 * recency_score +
0.10 * user_pinned +
0.10 * relation_value -
0.05 * sensitivity_risk
```

其中 `recency_score` 可以使用指数衰减：

```python
recency_score = exp(-days_since_last_access / half_life_days)
```

实现时，系统定期扫描候选记忆，跳过用户置顶、合规要求保留和高重要性记忆；对低分记忆执行删除、压缩或归档。伪代码：

```python
def smart_forget(memories, threshold=0.35):
    forgotten = []
    for memory in memories:
        if memory.pinned or memory.legal_hold:
            continue
        score = retention_score(memory)
        if score < threshold:
            forgotten.append(memory.id)
    return forgotten
```

记忆归档机制不是直接删除，而是把低访问、高潜在价值的记忆转移到冷存储，例如对象存储、压缩数据库或低成本向量索引。归档前应生成摘要和元数据，保留可检索索引：

```text
热存储：最近使用、高重要性、当前任务相关记忆
温存储：近期不常用但可能召回的记忆
冷存储：长期不用、低频访问、保留价值较高的历史记忆
```

四种记忆的集成方式不同。工作记忆一般不归档，过期后删除或整合。情景记忆适合归档为时间线摘要，例如“2026 年 6 月健康事件摘要”。语义记忆应谨慎归档，因为稳定知识可能长期有用；可把低置信度或低访问实体转入冷图谱。感知记忆通常体积大，例如图片、音频、传感器原始数据，应优先归档原始文件，只保留摘要、特征和引用 ID。

敏感信息删除不能只删业务数据库。向量数据库中可能保存了文本 embedding，图数据库中可能保存了实体和关系，日志、缓存、备份、对象存储和模型上下文记录也可能残留信息。彻底删除应包括：

1. 删除原始文本、附件和对象存储文件。
2. 删除向量数据库中的向量、payload 和元数据。
3. 删除图数据库中的实体、关系和相关属性。
4. 删除搜索索引、缓存、队列消息和临时文件。
5. 标记备份中的删除请求，按合规策略在备份过期或重建时清除。
6. 删除或脱敏日志中的敏感字段。
7. 如果敏感信息参与了摘要或整合记忆，要重新生成不含该信息的摘要。

对于“被遗忘权”场景，应维护 deletion tombstone，防止后续从归档或备份恢复时把已删除数据重新写回系统。

## 习题 8.4：学习助手中的 RAG 与 Memory 路由

在8.4节的“智能学习助手”案例中，我们结合了MemoryTool和RAGTool。请深入分析：

- 案例中的`ask_question()`方法同时使用了RAG检索和记忆检索。请分析：在什么情况下应该优先使用RAG？在什么情况下应该优先使用Memory？如何设计一个“智能路由”机制来自动选择最合适的检索方式？
- 当前的学习报告（`generate_report()`）只包含统计信息。请扩展这个功能，设计一个更智能的学习报告生成器：能够分析用户的学习轨迹、识别知识盲点、推荐下一步学习内容。这需要用到哪些记忆类型和检索策略？
- 假设你要将这个学习助手部署为多用户的Web服务，每个用户都有独立的记忆和知识库。请设计数据隔离方案：如何在Qdrant和Neo4j中实现用户级别的数据隔离？如何优化多用户场景下的检索性能？

### 参考答案

RAG 适合回答基于教材、文档、课程资料和外部知识的问题，例如“Transformer 的自注意力机制是什么”“第八章如何定义语义记忆”。这些问题需要稳定知识来源和引用证据。Memory 适合回答和用户个人学习过程有关的问题，例如“我上次哪里没学懂”“我最近总是错哪类题”“继续上次的学习计划”。这些问题需要调用用户历史行为、偏好、错误记录和学习轨迹。

智能路由可以先做意图分类，再决定检索组合：

```text
知识解释类 -> 优先 RAG
个人进度类 -> 优先 Memory
复习建议类 -> Memory + RAG
模糊问题 -> RAG 和 Memory 都检索，再由重排器选择
```

路由器可以输出结构化结果：

```json
{
  "route": "memory_rag",
  "reason": "用户询问下一步学习建议，需要结合个人薄弱点和课程知识",
  "memory_query": "最近错误题型和学习记录",
  "rag_query": "相关课程章节和练习内容"
}
```

更智能的学习报告应从“统计”升级为“诊断 + 建议”。报告可以包含：学习时间趋势、完成章节、正确率变化、高频错误概念、遗忘风险、兴趣偏好、知识图谱掌握情况、下一步学习计划。需要用到情景记忆记录每次学习事件和答题表现；语义记忆构建知识点图谱；工作记忆生成当前报告时保存中间分析；感知记忆可接入语音笔记、手写笔记或截图。检索策略上，Memory 检索用户历史错误和学习轨迹，RAG 检索对应教材解释和练习题，图检索找出前置知识和后续知识，向量检索找相似错题。

多用户 Web 服务必须做用户级隔离。在 Qdrant 中可以采用两种方案：小规模或强隔离场景使用每个用户一个 collection；大规模场景使用共享 collection，但每条向量 payload 中写入 `tenant_id` 或 `user_id`，查询时强制加 filter。共享 collection 更易扩展，但必须在服务端封装查询接口，不能让客户端直接传任意 filter。

在 Neo4j 中，可以为节点和关系增加 `tenant_id` 属性，所有 Cypher 查询都必须带 `WHERE n.tenant_id = $tenant_id`；企业版也可考虑多数据库隔离。为了防止越权，建议把查询构造封装在服务端 DAO 层，不允许 LLM 直接生成无约束 Cypher 访问全库。

性能优化包括：按用户或租户建立索引；高频用户使用缓存；向量检索先按 `tenant_id` 过滤再召回；大租户可独立 collection；冷热数据分层；对长历史先摘要再检索；批量写入学习记录；异步更新知识图谱。所有检索、写入和删除都应记录审计日志，便于排查跨用户数据泄露。

## 习题 8.5：语义记忆与知识图谱质量

语义记忆使用了Neo4j图数据库来存储知识图谱。请思考：

- 在8.2.5节的语义记忆实现中，系统会自动提取实体和关系构建知识图谱。请分析：这种自动提取的准确性如何？在什么情况下可能会提取出错误的实体或关系？如何设计一个“知识图谱质量评估”机制？
- 知识图谱的一个重要优势是支持复杂的关系推理。请设计一个查询场景，充分利用Neo4j的图查询能力（如多跳关系、路径查找），实现纯向量检索无法完成的任务。
- 对比语义记忆的“向量检索+图检索”混合策略与纯向量检索：在什么类型的查询中，图检索能够带来显著的性能提升？请用具体例子说明。

### 参考答案

自动实体关系提取的准确性取决于文本质量、领域术语、模型能力和 schema 约束。通用文本中常见实体和关系较容易提取，但在专业领域、缩写较多、指代复杂、句子很长或上下文不完整时容易出错。例如“苹果发布新芯片”中的“苹果”应识别为公司而不是水果；“Transformer 改进了 RNN 在长距离依赖上的不足”可能被错误抽成“Transformer 属于 RNN”；医学文本中药品、疾病、症状和禁忌关系如果抽错会带来严重风险。

知识图谱质量评估机制应包括自动校验和人工抽检。自动校验可以检查：实体类型是否合法，关系类型是否在 schema 内，置信度是否超过阈值，同一实体是否重复，关系方向是否合理，是否与已有高置信知识冲突，是否有来源文本证据。人工抽检适合高风险领域和低置信三元组。每条边应保存 `source_text`、`source_doc_id`、`confidence`、`created_at` 和 `extractor_version`，便于追溯和回滚。

可以设计评分：

```text
kg_quality_score =
0.30 * extraction_confidence +
0.25 * schema_validity +
0.20 * evidence_support +
0.15 * consistency_with_existing_graph +
0.10 * human_feedback
```

低于阈值的三元组进入待审核队列，不直接参与高风险推理。

一个充分利用 Neo4j 的查询场景是“为学习者推荐下一个最应该补的知识点”。图中有 `User`、`Concept`、`Exercise`、`Chapter` 节点，以及 `MASTERED`、`WEAK_AT`、`PREREQUISITE_OF`、`TESTS`、`BELONGS_TO` 等关系。系统可以查询：用户当前薄弱知识点的前置概念中，哪些尚未掌握，并且这些概念连接到最多后续知识点。

示例 Cypher 思路：

```cypher
MATCH (u:User {id: $user_id})-[:WEAK_AT]->(c:Concept)
MATCH (pre:Concept)-[:PREREQUISITE_OF*1..2]->(c)
WHERE NOT (u)-[:MASTERED]->(pre)
RETURN pre.name, count(c) AS affected_concepts
ORDER BY affected_concepts DESC
LIMIT 5
```

纯向量检索很难完成这种多跳前置关系推理，因为它只能找语义相似文本，不知道“谁是谁的前置知识”“补哪个点影响最大”。

混合策略比纯向量检索更适合关系密集型查询、依赖路径查询、因果链查询、推荐路径查询和需要精确实体约束的查询。例如用户问“我总是做错 LangGraph 的条件边，是不是因为前面哪个概念没掌握？”纯向量检索可能召回 LangGraph 条件边相关段落；图检索可以沿着知识点关系找到“状态机”“有向图”“条件分支”“状态对象”这些前置概念，再结合用户错题记录判断最可能的薄弱点。

图检索显著提升的典型问题包括：

```text
“A 和 B 之间通过哪些概念关联？”
“学习 C 之前必须掌握哪些前置知识？”
“某个错误模式影响哪些后续章节？”
“这个药物和用户已有疾病之间是否存在禁忌关系？”
“某客户、订单、工单、退款之间的完整链路是什么？”
```

这些问题不仅要求语义相似，还要求关系准确、路径可解释和实体约束明确，因此适合向量检索召回相关文本，图检索完成结构化推理。
