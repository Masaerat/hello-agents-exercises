# 第四章课后习题：智能体经典范式构建

## 习题 4.1：比较 ReAct、Plan-and-Solve 与 Reflection

本章介绍了三种经典智能体范式：`ReAct`、`Plan-and-Solve` 和 `Reflection`。请分析：

- 这三种范式在“思考”与“行动”的组织方式上有什么本质区别？
- 如果要设计一个“智能家居控制助手”（需要控制灯光、空调、窗帘等多个设备，并根据用户习惯自动调节），你会选择哪种范式作为基础架构？为什么？
- 是否可以将这三种范式组合使用？如果可以，请设计一个混合范式的智能体架构，并说明适用场景。

### 参考答案

ReAct 的核心是把推理和行动交织在同一个循环中：模型先产生 `Thought`，再选择 `Action` 调用工具，随后接收 `Observation`，再进入下一轮推理。它适合需要边查边想、根据外部反馈逐步推进的任务，例如搜索、问答、工具调用和排障。

Plan-and-Solve 的核心是先规划再执行。它会先把复杂任务拆成多个步骤，然后按计划逐步完成。它的优势是全局结构更清晰，适合多步骤、目标明确、依赖关系较强的任务；缺点是如果计划一开始有误，后续执行可能被错误计划牵引。

Reflection 的核心是执行后反思，通过“生成结果 -> 评估反馈 -> 修改优化”的循环提升质量。它适合输出质量要求高、可以迭代改进的任务，例如代码生成、文章写作、方案评审和复杂推理校验。它的缺点是成本更高，且需要合理的终止条件。

智能家居控制助手可以以 ReAct 作为基础架构，因为它需要持续感知环境并调用多个设备工具。例如用户说“我准备睡觉了”，智能体需要读取时间、温度、光照、用户习惯，然后调用灯光、空调、窗帘等工具。这个过程具有强交互性和实时反馈，适合 ReAct 的“思考-行动-观察”循环。

更完整的方案可以采用混合架构：Plan-and-Solve 用于复杂场景规划，例如“离家模式”“睡眠模式”“节能模式”；ReAct 用于每一步设备控制和状态确认；Reflection 用于在用户反馈不好时复盘策略，例如“昨晚空调太冷”后调整睡眠温度偏好。这种混合范式适合需要长期个性化、多设备协同和持续优化的智能家居系统。

## 习题 4.2：提升 ReAct 输出解析的鲁棒性

在 4.2 节的 `ReAct` 实现中，我们使用正则表达式解析大语言模型输出，例如 `Thought` 和 `Action`。请思考：

- 当前解析方法有哪些潜在脆弱性？在什么情况下可能失败？
- 除了正则表达式，还有哪些更鲁棒的输出解析方案？
- 尝试修改本章代码，使用一种更可靠的输出格式，并对比两种方案的优缺点。

### 参考答案

正则表达式解析的主要问题是对格式非常敏感。只要模型输出多了空格、换行、解释性文字、Markdown 代码块，或者把 `Action:` 写成中文“行动：”，解析就可能失败。模型还可能一次输出多个 Action，或者 Action 参数中包含括号、引号、逗号等复杂字符，导致正则匹配错误。更严重的是，如果模型生成了看似合法但参数不完整的工具调用，正则只能提取文本，无法验证语义和类型。

更鲁棒的方案包括 JSON 输出、函数调用接口、结构化 schema 校验、Pydantic/BaseModel 解析、XML 标签格式、有限状态机解析，以及使用模型服务原生的 tool calling/function calling 能力。工程上更推荐“结构化输出 + schema 校验 + 重试修复”的组合。

例如可以要求模型只输出 JSON：

```json
{
  "thought": "我需要先查询天气",
  "action": {
    "tool": "get_weather",
    "arguments": {
      "city": "北京"
    }
  }
}
```

系统拿到 JSON 后先解析，再检查 `tool` 是否在允许列表中，`arguments` 是否符合该工具的参数 schema。如果解析失败，可以把错误信息反馈给模型，让它按格式重新输出。

正则方案的优点是简单、容易快速演示，不需要复杂依赖；缺点是脆弱、难验证、难扩展。JSON/schema 方案实现成本略高，但可维护性、可观测性和安全性更好，特别适合生产系统和多工具智能体。

## 习题 4.3：扩展 ToolExecutor 与工具选择机制

工具调用是现代智能体的核心能力之一。基于 4.2.2 节的 `ToolExecutor` 设计，请完成以下扩展实践：

- 为 `ReAct` 智能体添加一个“计算器”工具，使其能够处理复杂数学计算问题，例如“计算 `(123 + 456) × 789 / 12 = ?` 的结果”。
- 设计并实现一个“工具选择失败”的处理机制：当智能体多次调用错误工具或提供错误参数时，系统应如何引导它纠正？
- 如果可调用工具增加到 50 个甚至 100 个，当前工具描述方式是否仍然有效？从工程角度如何优化工具组织和检索机制？

### 参考答案

计算器工具可以设计为接收一个数学表达式字符串，返回计算结果。生产系统中不应直接对用户输入执行任意 `eval`，更安全的做法是使用受限表达式解析器，只允许数字、括号和基础运算符。示例接口可以是：

```python
def calculator(expression: str) -> str:
    """计算安全数学表达式，只允许加减乘除和括号。"""
    result = safe_eval_math_expression(expression)
    return str(result)
```

对于 `(123 + 456) * 789 / 12`，先算 `123 + 456 = 579`，再算 `579 * 789 = 456831`，最后 `456831 / 12 = 38069.25`，因此结果是 `38069.25`。

工具选择失败机制可以包含三层。第一层是参数校验：工具执行前检查工具名是否存在、必填参数是否齐全、参数类型是否正确。第二层是错误反馈：如果调用失败，把结构化错误作为 Observation 返回给智能体，例如“工具 calculator 需要参数 expression，但当前缺失”。第三层是失败上限和纠偏：如果连续 2-3 次工具选择失败，系统应要求模型重新阅读工具列表，或触发一个专门的“工具选择器”步骤；如果仍失败，应停止自动执行并请求用户或人工介入。

当工具数量增加到 50 或 100 个时，把所有工具描述都塞进提示词会变得低效且不可靠。模型可能忽略关键工具，混淆相似工具，提示词成本也会显著上升。更好的工程方案是工具分组、按任务动态检索工具、为工具建立向量索引、使用工具路由器、分层工具目录和权限控制。流程可以是：先根据用户任务检索候选工具 3-8 个，再让智能体在候选工具中选择；对高风险工具增加确认和审计。

## 习题 4.4：Plan-and-Solve 的动态重规划

`Plan-and-Solve` 范式将任务分解为“规划”和“执行”两个阶段。请分析：

- 在 4.3 节实现中，规划阶段生成的计划是静态的。如果执行过程中发现某个步骤无法完成或结果不符合预期，应该如何设计“动态重规划”机制？
- 对比 `Plan-and-Solve` 与 `ReAct`：在处理“预订一次从北京到上海的商务旅行（包括机票、酒店、租车）”这样的任务时，哪种范式更合适？为什么？
- 尝试设计一个“分层规划”系统：先生成高层抽象计划，然后针对每个高层步骤生成详细子计划。这种设计有什么优势？

### 参考答案

动态重规划机制需要让计划不再是一份一次性静态清单，而是一个可更新的状态对象。每个步骤应包含目标、依赖、状态、执行结果和失败原因。当某一步失败时，执行器把失败信息写入状态，例如“目标酒店满房”“航班价格超预算”“租车门店不可用”。随后触发重规划器，基于当前已完成步骤、失败原因、用户约束和剩余目标生成新的计划。重规划可以只修改受影响的局部步骤，不必完全推翻整个计划。

商务旅行预订更适合以 Plan-and-Solve 为主，因为它包含多个相互依赖的子任务：确认时间、订机票、订酒店、安排租车、核对预算、生成行程。先规划能让系统明确依赖关系，例如航班到达时间会影响酒店入住和租车时间。不过执行每一步时仍然需要 ReAct，因为查询机票、比较酒店、调用预订 API 都需要根据外部结果不断调整。因此最佳方案通常是 Plan-and-Solve 做全局任务分解，ReAct 做每个步骤的工具执行与反馈处理。

分层规划可以分为高层计划和子计划。高层计划例如：

```text
1. 确认出行约束
2. 安排交通
3. 安排住宿
4. 安排本地出行
5. 汇总并确认行程
```

每个高层步骤再展开子计划。例如“安排交通”可以拆成查询航班、比较价格和时间、确认乘机人信息、预订、记录票号。分层规划的优势是结构清晰、便于局部重规划、便于人类审核，也能降低一次性生成超长计划的错误率。它还适合大型任务中的多智能体分工：高层规划器负责目标分解，子任务执行器负责具体操作。

## 习题 4.5：Reflection 的成本、模型分工与终止条件

`Reflection` 机制通过“执行-反思-优化”循环来提升输出质量。请思考：

- 在 4.4 节代码生成案例中，不同阶段使用的是同一个模型。如果使用两个不同模型（例如更强模型做反思，更快模型做执行），会带来什么影响？
- `Reflection` 的终止条件是“反馈中包含无需改进”或“达到最大迭代次数”。这种设计是否合理？能否设计更智能的终止条件？
- 假设要搭建一个“学术论文写作助手”，它能够生成初稿并不断优化论文内容。请设计一个多维度 Reflection 机制，从段落逻辑性、方法创新性、语言表达、引用规范等角度进行反思和改进。

### 参考答案

使用两个不同模型可以形成“快模型执行、强模型评审”的分工。执行模型负责快速生成初稿或代码，成本低、速度快；反思模型负责发现问题、提出修改建议，要求更强的推理和评审能力。这样可以在成本和质量之间取得平衡。缺点是系统复杂度增加，不同模型的风格和判断标准可能不一致，反思模型给出的建议也可能超出执行模型的修改能力。

“反馈中包含无需改进”或“达到最大迭代次数”是一个可用但偏粗糙的终止条件。它的问题是过度依赖模型自我判断，可能过早停止，也可能在收益很小的情况下继续迭代。更智能的终止条件可以结合多种信号：质量评分超过阈值、连续两轮改进幅度很小、没有高优先级问题、关键测试通过、事实校验通过、预算或时间接近上限、用户确认满意。对于代码任务，还可以把单元测试、lint、类型检查结果作为终止依据。

学术论文写作助手可以采用多维度 Reflection。初稿生成后，系统分别从以下维度反思：段落逻辑是否连贯，研究问题是否明确，方法是否有创新性，实验设计是否支撑结论，语言是否准确简洁，引用是否规范，是否存在夸大结论或缺少证据的问题。每个维度可以由独立的评审提示词或评审 agent 处理，输出结构化反馈：

```text
维度：引用规范
问题：第 3 段提到某方法优于基线，但没有引用来源。
建议：补充对应论文引用，或改写为更保守的表述。
优先级：高
```

优化器再根据反馈修改论文，并保留修改记录。高风险内容如实验结论、引用和相关工作比较，应要求提供证据来源，不能由模型自由编造。最终版本最好由人类作者确认。

## 习题 4.6：提示词工程与范式行为

提示词工程是影响智能体最终效果的关键技术。本章展示了多个提示词模板。请分析：

- 对比 4.2.3 节的 `ReAct` 提示词和 4.3.2 节的 `Plan-and-Solve` 提示词，它们在结构设计上有什么不同？这些差异如何服务于各自范式的核心逻辑？
- 在 4.4.3 节的 `Reflection` 提示词中，使用了“你是一位极其严格的代码评审专家”这样的角色设定。尝试修改角色设定，观察输出变化，并总结角色设定对智能体行为的影响。
- 在提示词中加入 `few-shot` 示例通常能提升模型对特定格式的遵循能力。请为本章某个智能体添加 `few-shot` 示例，并对比其效果。

### 参考答案

ReAct 提示词通常强调循环格式，例如 `Thought -> Action -> Observation`，并明确可用工具和 Action 格式。它的重点是让模型每轮只做一个可执行动作，并根据观察结果继续推理。因此 ReAct 提示词必须严格约束工具调用格式、停止条件和每轮输出。

Plan-and-Solve 提示词更强调先完整规划，再逐步执行。它通常要求模型把任务拆分为编号步骤，明确每一步目标和依赖，然后执行器按计划推进。它服务的是全局分解能力，而不是每轮即时行动。因此 Plan-and-Solve 的提示词更关注计划完整性、步骤顺序和任务覆盖。

Reflection 提示词中的角色设定会显著影响模型关注点。“极其严格的代码评审专家”会倾向指出更多问题，关注边界情况、质量缺陷和潜在风险；如果改为“注重代码可读性的开源项目维护者”，输出可能更关注命名、结构、注释、风格一致性和维护成本；如果改为“性能优化专家”，则会更关注复杂度、缓存、I/O 和瓶颈。角色设定本质上是在调整模型的评价标准和输出偏好。

few-shot 示例可以帮助模型稳定遵循格式。例如给 ReAct 智能体加入一个示例：

```text
用户：北京今天适合去哪里玩？
Thought: 我需要先查询北京天气。
Action: get_weather(city="北京")
Observation: 北京今天晴，微风。
Thought: 晴天适合户外景点，我需要推荐合适地点。
Action: search_attraction(city="北京", preference="户外")
```

加入 few-shot 后，模型更容易按指定格式输出，减少漏写 `Action`、一次输出多个动作或自由发挥的问题。缺点是提示词变长，成本增加，示例选择不当还可能让模型过度模仿示例而降低泛化能力。

## 习题 4.7：设计电商客服智能体

某电商初创公司希望使用“客服智能体”代替部分真人客服以降本增效。系统需要具备以下功能：

- 理解用户的退款申请理由。
- 查询用户的订单信息和物流状态。
- 根据公司政策判断是否应该批准退款。
- 生成一封得体的回复邮件并发送至用户邮箱。
- 如果判断存在争议（自我置信度低于阈值），能够进行自我反思并给出更审慎的建议。

作为产品负责人，请回答：

- 你会选择本章哪种范式或哪些范式的组合作为核心架构？
- 这个系统需要哪些工具？请列出至少 3 个工具及其功能描述。
- 如何设计提示词，确保智能体决策既符合公司利益，又能保持对用户的友好态度？
- 产品上线后可能面临哪些风险和挑战？如何通过技术手段降低风险？

### 参考答案

这个系统适合使用混合范式：Plan-and-Solve 负责把退款处理拆成步骤，ReAct 负责在每一步调用订单、物流、政策和邮件工具，Reflection 负责在低置信度或高风险案例中复核判断。整体流程可以是：理解用户诉求 -> 查询订单和物流 -> 检索退款政策 -> 形成初步判断 -> 如果置信度低则反思或转人工 -> 生成回复邮件 -> 用户或系统确认后发送。

系统至少需要以下工具：

- `get_order(order_id)`：查询订单金额、商品类型、购买时间、支付状态、用户信息。
- `get_logistics(order_id)`：查询发货、签收、退货物流和异常状态。
- `search_refund_policy(product_type, days_since_purchase)`：检索适用的退款政策。
- `risk_check(user_id, order_id)`：检查异常退款频率、疑似欺诈等风险信号。
- `draft_email(user_id, decision, reason)`：生成客服邮件草稿。
- `send_email(to, subject, body)`：发送邮件。该工具应设置人工确认或高风险拦截。

提示词应同时强调政策边界和用户体验。例如：

```text
你是电商售后客服智能体。你的目标是在遵守公司退款政策和法律要求的前提下，给用户提供清晰、礼貌、可执行的解决方案。
决策时必须基于订单信息、物流状态和退款政策，不得编造事实。
如果证据不足、政策不明确、金额较高或置信度低，必须转人工审核。
回复用户时应说明原因、给出下一步操作，并保持友好语气。
```

上线风险包括：错误退款导致公司损失，错误拒绝导致用户投诉，模型幻觉导致政策误用，工具调用越权，邮件误发，用户隐私泄露，恶意用户诱导模型绕过政策，以及低置信度场景处理不当。技术缓解手段包括：权限控制、工具参数校验、结构化输出、政策检索增强、金额和风险阈值、人工审核、操作日志、邮件发送前确认、敏感信息脱敏、离线评测集、灰度发布和持续监控。对于实际退款或发送邮件这类有副作用的操作，应默认加入确认机制和可追溯记录。
