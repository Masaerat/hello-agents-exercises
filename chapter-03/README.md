# 第三章课后习题：大语言模型基础

## 习题 3.1：从 N-gram 到 Transformer

自然语言处理中，语言模型经历了从统计模型到神经网络模型的演进。请回答：

- 使用本章提供的迷你语料库 `datawhale agent learns`、`datawhale agent works`，计算句子 `agent works` 在 Bigram 模型下的概率。
- N-gram 模型的核心假设是马尔可夫假设。请解释这个假设的含义，以及 N-gram 模型有哪些根本性局限。
- 神经网络语言模型（RNN/LSTM）和 Transformer 分别如何克服 N-gram 模型的局限？它们各自的优势是什么？

### 参考答案

迷你语料库包含两个句子：

```text
datawhale agent learns
datawhale agent works
```

在 Bigram 模型中，句子 `agent works` 的概率可以写成：

```text
P(agent works) = P(agent) * P(works | agent)
```

如果只计算条件转移部分，通常关注：

```text
P(works | agent) = Count(agent works) / Count(agent)
```

语料中 `agent` 出现 2 次，`agent works` 出现 1 次，因此：

```text
P(works | agent) = 1 / 2 = 0.5
```

如果加入句首标记 `<s>`，两个句子都以 `datawhale` 开头，则 `P(agent | <s>)` 不是直接可得，因为 `agent` 并不在句首。若把 `agent works` 当作一个独立片段并只考察从 `agent` 到 `works` 的 Bigram 概率，答案就是 0.5。若要严格计算完整句子的生成概率，应先明确句首、句尾标记和平滑方法。

马尔可夫假设指的是：预测当前词时，只依赖前面有限个词，而不依赖完整历史。Bigram 只依赖前一个词，Trigram 只依赖前两个词。这一假设让统计建模变得简单，但也带来明显局限：上下文窗口短，无法表达长距离依赖；数据稀疏严重，没见过的词组合概率可能为 0；无法理解语义，只是统计局部共现；对同义表达、复杂句法和开放域文本泛化能力弱。

RNN/LSTM 使用隐藏状态把历史信息压缩到连续向量中，能够在理论上利用更长上下文，并通过词向量缓解稀疏性。LSTM 通过门控机制改善了普通 RNN 的长期依赖和梯度消失问题。Transformer 则通过自注意力机制让序列中任意位置的 token 可以直接建立联系，避免了 RNN 必须按时间步串行传递信息的问题。Transformer 的优势是并行训练效率高、长距离依赖建模能力强、可扩展性好，因此成为现代大语言模型的核心架构。

## 习题 3.2：理解 Transformer 架构

Transformer 架构是现代大语言模型的基础。请回答：

- 自注意力机制（Self-Attention）的核心思想是什么？
- 为什么 Transformer 能够并行处理序列，而 RNN 必须串行处理？位置编码在其中起什么作用？
- Decoder-Only 架构与完整的 Encoder-Decoder 架构有什么区别？为什么现在主流的大语言模型多采用 Decoder-Only 架构？

### 参考答案

自注意力机制的核心思想是：序列中的每个 token 在生成自己的表示时，都可以根据相关性动态关注同一序列中的其他 token。具体来说，每个 token 会被映射为 Query、Key、Value。模型用 Query 和 Key 计算注意力权重，再用这些权重对 Value 加权求和，得到融合上下文信息的新表示。这样，模型不需要固定窗口，也不需要一步步传递隐藏状态，就能直接建模 token 之间的依赖关系。

RNN 必须串行处理，是因为第 `t` 个位置的隐藏状态依赖第 `t-1` 个位置的隐藏状态，后一步必须等待前一步完成。Transformer 的自注意力在训练时可以同时看到整个输入序列，所有位置的 Query、Key、Value 可以一次性矩阵化计算，因此更适合 GPU 并行。位置编码的作用是补充顺序信息，因为自注意力本身对 token 顺序不敏感。如果没有位置编码，模型很难区分“猫追狗”和“狗追猫”这类词相同但顺序不同的句子。

Encoder-Decoder 架构包含编码器和解码器。编码器负责理解输入序列，解码器基于编码器输出逐步生成目标序列，典型用于机器翻译等输入输出都明确的任务。Decoder-Only 架构只保留解码器，用因果掩码保证模型只能看到当前位置之前的 token，从而执行自回归生成。

主流大语言模型多采用 Decoder-Only 架构，是因为它与“预测下一个 token”的预训练目标天然匹配，结构统一，适合大规模无监督文本训练，也能通过提示词同时处理问答、续写、摘要、代码生成、工具调用等多种任务。相比为不同任务设计不同输入输出结构，Decoder-Only 更简洁、可扩展，也更符合通用生成模型的需求。

## 习题 3.3：字符、单词与 BPE 分词

文本子词分词算法是大语言模型的一项关键技术，负责将文本转换为模型可处理的 token 序列。请回答：为什么不能直接以“字符”或“单词”作为模型输入单元？BPE（Byte Pair Encoding）算法解决了什么问题？

### 参考答案

直接使用字符作为输入单元的优点是词表很小，几乎不会遇到未登录词。但缺点是序列会变得很长，模型需要更多步才能表达一个词或短语，训练和推理成本更高，也更难学习词级语义。例如英文单词 `unbelievable` 如果按字符切分，模型需要从许多字符组合中学习词义。

直接使用单词作为输入单元的优点是语义粒度清晰，序列较短。但缺点是词表会非常大，尤其在英文变形、中文词组、代码标识符、专有名词、多语言文本中会出现大量低频词和未登录词。未见过的新词如果不在词表中，模型无法直接处理，只能映射为 `<unk>`，这会损失信息。

BPE 的目标是在字符和单词之间取得平衡。它从字符级别开始，反复合并语料中最常见的相邻符号对，形成子词单元。例如 `low`、`lower`、`lowest` 可能共享 `low` 这个子词。这样既能用有限词表覆盖开放词汇，又能让常见词或常见片段以较短 token 表示。BPE 缓解了未登录词问题，降低了词表规模，同时保留了一定的语义和形态结构，因此成为大语言模型常用的分词思路之一。

## 习题 3.4：本地部署开源模型与提示策略

本章介绍了如何本地部署开源大语言模型。请完成以下实践和分析：

- 按照本章指导，在本地部署一个轻量级开源模型（例如 Qwen3-0.6B），并尝试调整采样参数，观察它们对输出的影响。
- 选择一个具体任务，例如文本分类、信息抽取、代码生成等，设计并对比 Zero-shot、Few-shot、Chain-of-Thought 等提示策略的效果差异。
- 从性能、成本、可控性、隐私等维度比较闭源模型和开源模型。
- 如果要构建一个企业级客服智能体，你会选择哪种类型的模型？需要考虑哪些因素？

### 参考答案

采样参数会显著影响模型输出。较低的 temperature 会让输出更稳定、保守、重复性更高，适合事实问答、分类、信息抽取等任务；较高的 temperature 会增加随机性和创造性，适合头脑风暴、文案生成等任务。top-p 控制候选 token 的概率质量范围，较低 top-p 会收窄候选空间，较高 top-p 会允许更多可能表达。max tokens 控制输出长度，过短可能截断答案，过长可能导致跑题或成本增加。

以“文本分类”为例，Zero-shot 提示可以直接要求模型判断文本类别，不提供示例，优点是简单，缺点是模型可能误解标签边界。Few-shot 提示给出几个输入和标签示例，能帮助模型对齐分类标准，通常比 Zero-shot 更稳定。Chain-of-Thought 让模型先分析理由再给结论，适合复杂判断，但在简单分类中可能增加成本和不必要的输出。实际系统中可以要求模型“先内部分析，最终只输出标签”，以兼顾推理和结构化结果。

闭源模型通常性能更强、部署简单、更新快，适合快速验证和对效果要求高的场景；缺点是调用成本受供应商定价影响，数据需要发送到外部服务，可控性和可解释性有限。开源模型的优势是可本地部署、隐私可控、可微调、成本结构更可控，也便于针对垂直领域优化；缺点是需要硬件资源和工程维护，模型效果、推理速度和安全能力可能弱于顶级闭源模型。

企业级客服智能体的模型选择应综合考虑效果、成本、延迟、隐私、合规、可用性、可观测性、可微调能力和工具调用能力。如果业务涉及敏感客户数据或强合规要求，可以优先考虑私有化部署的开源模型，配合检索增强和人工兜底；如果业务更重视快速上线和强通用能力，可以选择闭源模型 API，但要做好数据脱敏、权限控制、日志审计和供应商风险评估。实际落地常采用混合方案：高频简单问题用小模型或规则，复杂问题用强模型，敏感或高风险问题转人工。

## 习题 3.5：模型幻觉与缓解方法

模型幻觉是大语言模型当前存在的关键局限之一。本章介绍了检索增强生成、多步推理、外部工具调用等缓解方法。请回答：

- 选择其中一种方法，说明其工作原理和适用场景。
- 调研是否还有其他缓解模型幻觉的方法，它们有哪些改进和优势？

### 参考答案

以检索增强生成（RAG）为例，它的核心思路是：不要只依赖模型参数中的记忆，而是在回答前先从外部知识库、文档库或搜索系统中检索相关内容，再把检索结果作为上下文提供给模型生成答案。这样，模型的回答可以被具体资料约束，减少凭空编造的概率。RAG 特别适合企业知识库问答、论文阅读、法律法规查询、产品客服、内部文档助手等知识经常更新或要求引用来源的场景。

RAG 的关键组件包括文档切分、向量化、索引、召回、重排序、上下文拼接和带引用生成。它能缓解模型知识过时和事实不可靠的问题，但并不能完全消除幻觉。如果检索结果不相关、文档本身错误、上下文过长或模型没有正确使用证据，仍然可能产生错误答案。

其他缓解方法包括外部工具调用、事实校验、约束解码、结构化输出、模型自检、多智能体交叉审查和人工反馈。外部工具调用适合计算、查询实时数据、访问数据库等任务，让模型把不可靠的“猜测”变成可验证的工具结果。事实校验可以在生成后用另一个模型或规则系统检查关键声明是否有证据支持。结构化输出和 schema 校验能减少格式错误和越界回答。多智能体交叉审查可以让一个 agent 生成答案，另一个 agent 专门找证据和漏洞。对于高风险场景，最可靠的做法通常是组合使用：RAG 提供依据，工具调用验证事实，规则和人工审核约束最终输出。

## 习题 3.6：设计论文辅助阅读智能体

假设你要设计一个论文辅助阅读智能体，它能够帮助研究人员快速阅读并理解学术论文，包括总结论文核心内容、回答关于论文的问题、提取关键信息、比较多篇论文的观点等。请回答：

- 你会选择哪个模型作为基座模型？选择时需要考虑哪些因素？
- 如何设计提示词来引导模型更好地理解学术论文？如果论文超过模型上下文窗口限制，你会如何解决？
- 学术研究强调严谨性。系统中应加入哪些设计，以确保生成信息准确、客观、忠于原文？

### 参考答案

基座模型应选择长文本理解能力强、推理能力稳定、支持结构化输出、工具调用能力好、中文和英文论文都表现可靠的模型。如果论文涉及专业领域，还要考虑模型是否具备相关领域知识，是否支持私有化部署，成本和延迟是否可接受。对于企业或实验室内部论文系统，如果涉及未公开论文或敏感研究数据，隐私和部署方式也很关键。可以选择强闭源模型做高质量分析，也可以选择较强开源模型本地部署，并通过检索增强补足知识和上下文。

提示词设计应要求模型围绕论文结构进行阅读，例如标题、摘要、问题定义、方法、实验设置、主要结果、贡献、局限和可复现性。可以要求输出固定结构：

```text
1. 论文要解决的问题
2. 核心方法
3. 关键实验和结论
4. 与已有工作的区别
5. 局限性
6. 可追问的问题
```

对问答任务，应明确要求“只基于论文原文回答；如果原文没有提到，请回答无法从原文确定”。对多篇论文比较，应要求模型分别提取每篇论文的观点，再进行对齐比较，避免混合不同论文的信息。

如果论文超过上下文窗口，可以采用分块阅读和层级总结。先按章节或语义段落切分论文，对每块提取关键信息和引用位置；再把各块摘要汇总成论文级摘要；问答时通过检索找到相关块，再让模型基于这些块回答。对于公式、表格和图，可以用专门解析工具提取结构化信息，避免直接丢失关键内容。

为了保证准确客观，系统应加入证据引用、原文定位、检索增强、事实一致性检查和人工确认机制。每个关键结论最好附带页码、段落或原文片段来源。模型不应编造实验结果或引用，无法确认时应明确说明不确定。对多篇论文比较，应保留每个观点来自哪篇论文。对于最终报告，可以加入“结论-证据”对应表，让研究人员快速检查模型说法是否忠于原文。
