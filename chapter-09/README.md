# 第九章课后习题：上下文工程

> <strong>提示</strong>：部分习题没有标准答案，重点在于培养学习者对上下文工程和长时程任务管理的综合理解和实践能力。

## 习题 9.1：上下文工程、上下文腐蚀与系统提示平衡

本章介绍了上下文工程与提示工程的区别。请分析：

- 在9.1节中提到“上下文必须被视作一种有限资源，且具有边际收益递减”。请解释什么是“上下文腐蚀”（context rot）现象？为什么即使模型支持100K甚至200K的上下文窗口，我们仍然需要谨慎管理上下文？
- 假设你要构建一个“代码审查助手”，需要分析一个包含50个文件的代码库。请对比两种策略：（1）一次性将所有文件内容加载到上下文中；（2）使用JIT（Just-in-time）上下文，通过工具按需检索文件。分析各自的优缺点和适用场景。
- 在9.2.1节中提到系统提示的两个极端误区：“过度硬编码”和“过于空泛”。请各举一个实际例子，并说明如何找到合适的平衡点。

### 参考答案

“上下文腐蚀”指上下文窗口越来越长后，模型对关键信息的注意力和执行稳定性下降的现象。表现包括：早期重要约束被忽略，后加入的无关信息干扰判断，相似片段互相混淆，模型引用过期状态，或者在大量文本中抓住了错误线索。它不是单纯的“窗口不够大”，而是“上下文质量下降导致有效信息密度变低”。

即使模型支持 100K 或 200K 上下文，也仍然需要管理上下文。第一，长上下文成本更高，延迟更大。第二，模型并不会等价地利用所有 token，靠近当前位置、结构清晰、显式标注的信息更容易被使用。第三，长上下文中可能包含冲突信息、旧计划、失败尝试和无关细节，增加误判概率。第四，生产系统还要考虑隐私、权限、审计和可复现性，不能把所有内容无差别塞进模型。

代码审查助手如果一次性加载 50 个文件，优点是模型能看到完整代码，适合小型仓库、一次性全局架构审查、跨文件依赖非常密集且文件总量不大的场景。缺点是成本高、上下文噪声大、容易忽略关键 diff，且大仓库会超出窗口或导致响应慢。它还可能把不相关文件纳入推理，增加误报。

JIT 上下文策略是先读取目录结构、变更 diff、符号索引、测试结果，再按需要检索具体文件和函数。优点是上下文精简、相关性高、可逐步深入、适合大型仓库和长时程任务。缺点是需要工具支持和检索策略；如果首次检索漏掉关键依赖，模型可能基于不完整信息做判断。工程上更推荐 JIT：先加载高层地图和变更摘要，再按调用链、引用关系和测试失败点逐步展开。

过度硬编码的系统提示例子：

```text
你必须永远按 A、B、C 三步回答；每次都必须调用 search_tool；不允许提出任何澄清问题；所有代码审查都必须输出 10 条问题。
```

这种提示会让模型在不需要搜索时也搜索，在问题不足 10 条时编造问题，面对不清晰需求也无法澄清。

过于空泛的系统提示例子：

```text
你是一个优秀的助手，请尽力帮助用户。
```

它没有说明任务边界、输出格式、工具使用规则、风险处理方式，复杂任务中很难得到稳定行为。

合适的平衡点是“原则稳定、细节可适配”。系统提示应明确角色、目标、约束、工具边界、输出格式和高风险处理规则，但不要硬编码每个具体场景。例如代码审查助手可以规定“优先报告会导致错误、回归、安全风险的问题；必须给出文件和行号；不确定时说明假设；不要为了凑数量编造问题”，但不应规定每次必须输出固定数量的问题。

## 习题 9.2：GSSC 流水线与上下文质量评估

GSSC（Gather-Select-Structure-Compress）流水线是本章的核心技术。请深入思考：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- 在9.3节的ContextBuilder实现中，四个阶段各有不同的职责。请分析：如果某个阶段失效（如Select阶段选择了不相关的信息，或Compress阶段过度压缩导致信息丢失），会对最终的智能体表现产生什么影响？
- 请基于9.3.4节的代码，为ContextBuilder添加一个“上下文质量评估”功能：在每次构建上下文后，自动评估上下文的信息密度、相关性和完整性，并给出优化建议。
- GSSC流水线中的“压缩”阶段使用了LLM进行智能摘要。请思考：在什么情况下，简单的截断（truncation）或滑动窗口（sliding window）策略可能比LLM摘要更合适？设计一个混合压缩策略，结合多种压缩方法的优势。

### 参考答案

GSSC 四个阶段分别影响上下文质量。`Gather` 负责收集候选信息，如果收集不足，后续阶段无论多聪明都无法补齐缺失事实；如果收集过多，会增加选择和压缩压力。`Select` 负责筛选相关信息，如果选错，模型会基于无关材料回答，出现偏题或误判。`Structure` 负责组织上下文，如果结构混乱，模型难以区分任务目标、证据、历史状态和当前约束。`Compress` 负责压缩，如果过度压缩，会丢失关键细节，例如错误日志中的具体异常、代码中的边界条件；如果压缩不足，又会导致上下文冗长和成本上升。

上下文质量评估可以在构建完成后输出三个维度：信息密度、相关性、完整性。信息密度衡量有效信息占比，相关性衡量上下文与当前任务的匹配程度，完整性衡量是否覆盖完成任务所需的关键方面。示例接口：

```python
class ContextQualityReport(BaseModel):
    density_score: float
    relevance_score: float
    completeness_score: float
    missing_items: list[str]
    redundant_items: list[str]
    suggestions: list[str]

class ContextBuilder:
    def build(self, task: str) -> BuiltContext:
        gathered = self.gather(task)
        selected = self.select(task, gathered)
        structured = self.structure(task, selected)
        compressed = self.compress(task, structured)
        quality = self.evaluate_quality(task, compressed, gathered)
        return BuiltContext(content=compressed, quality=quality)
```

评估可以结合规则和模型。规则评估包括 token 占用、重复率、引用数量、是否包含任务目标、是否包含当前状态、是否包含关键错误日志。模型评估可以让 LLM 判断：“给定任务和上下文，完成任务还缺什么？”输出结构化建议。例如如果任务是修复测试失败，但上下文没有测试命令和错误堆栈，完整性分数应降低。

简单截断或滑动窗口在某些情况下比 LLM 摘要更合适。第一，最新信息最重要时，例如实时聊天、日志尾部、命令输出末尾的错误栈。第二，原文细节不可改写时，例如代码、配置、SQL、错误日志、法律条款。第三，对延迟和成本极敏感时，调用 LLM 摘要不划算。第四，摘要可能引入幻觉或丢失精确字段时，应保留原文片段。

混合压缩策略可以是：

```text
1. 对代码、错误日志、配置、数字表格：保留原文关键片段，使用滑动窗口或按行截断。
2. 对历史对话和探索过程：使用 LLM 摘要，保留决策、结论、未解决问题。
3. 对重复文档：先去重，再抽取标题、关键段落和引用。
4. 对长文档：先结构化分块，再检索相关块，只摘要低风险背景内容。
5. 对高风险事实：摘要后必须附原文引用或片段 ID。
```

这样既能减少 token，又能避免把需要精确保真的信息摘要坏。

## 习题 9.3：NoteTool、TerminalTool 与重构助手

NoteTool和TerminalTool是支持长时程任务的关键工具。基于9.4节和9.5节的内容，请完成以下扩展实践：

> <strong>提示</strong>：这是一道动手实践题，建议实际操作

- NoteTool使用了分层笔记系统（项目笔记、任务笔记、临时笔记）。请设计一个“笔记自动整理”机制：当临时笔记积累到一定数量时，智能体能够自动分析这些笔记，将重要信息提升为任务笔记或项目笔记，并清理冗余内容。
- TerminalTool提供了文件系统操作能力，但在9.5.2节中强调了安全性设计。请分析：当前的安全机制（路径验证、命令白名单、权限检查）是否足够？如果智能体需要访问敏感文件或执行危险操作，应该如何设计一个“人机协作审批”流程？
- 结合NoteTool和TerminalTool，设计一个“智能代码重构助手”：能够分析代码库结构、记录重构计划、逐步执行重构操作，并在笔记中追踪进度和遇到的问题。请画出完整的工作流程图。

### 参考答案

笔记自动整理可以采用“触发器 + 分类器 + 提炼器 + 清理器”的机制。当临时笔记数量超过阈值、token 总量超过阈值、任务阶段完成或会话即将结束时触发整理。系统先对临时笔记分类：事实、决策、待办、问题、命令结果、废弃尝试。然后判断哪些应提升为任务笔记，哪些应提升为项目笔记。

提升规则可以是：

```text
项目笔记：长期有效的信息，例如项目结构、架构约束、编码规范、重要命令、部署方式。
任务笔记：当前任务相关的信息，例如重构计划、已改文件、失败测试、待验证项。
临时笔记：短期探索信息，例如某次命令输出、一个已排除的猜测。
```

整理后，任务笔记应保留结论和证据引用，临时笔记可以删除重复项或标记为已归档。例如“尝试 A 失败，因为测试 X 报错”不应简单删除，而应压缩成“已排除 A，原因是 X”，避免未来重复踩坑。

路径验证、命令白名单和权限检查是必要基础，但并不足以覆盖所有风险。原因是合法命令也可能造成危险后果，例如 `git clean`、批量移动文件、读取密钥文件、向外部上传数据；白名单命令也可能通过参数执行高风险操作；路径验证也无法判断文件内容是否敏感。

人机协作审批流程应按风险分级：

```text
低风险：只读命令，如列目录、读取普通源码、搜索文本，可自动执行。
中风险：修改工作区文件、运行格式化、安装依赖，需要展示计划和影响范围，可一次性审批。
高风险：删除文件、覆盖大量文件、读取密钥、执行网络上传、改 git 历史，必须逐次审批。
禁止操作：绕过权限、读取明确无关的私密目录、执行无法审计的任意脚本。
```

审批请求应包含：操作命令、目标路径、影响文件列表、风险等级、回滚方案、为什么需要执行。执行后记录审计日志，包括审批人、时间、命令、输出摘要和变更 diff。

智能代码重构助手流程：

```text
用户提出重构目标
-> TerminalTool 读取目录结构、依赖文件、测试命令
-> NoteTool 写入项目笔记：项目结构、技术栈、约束
-> 分析目标模块和调用关系
-> NoteTool 写入任务笔记：重构计划、风险点、验收标准
-> 用户确认计划
-> TerminalTool 分步读取和修改文件
-> 每完成一小步：
   -> 运行相关测试或静态检查
   -> NoteTool 记录已完成项、失败项、下一步
   -> 如果失败，记录原因并回退或修复
-> 全量验证
-> NoteTool 整理最终总结：改动文件、验证结果、遗留风险
```

流程图：

```text
Start
-> Explore Repo
-> Write Project Notes
-> Build Refactor Plan
-> Write Task Notes
-> Approve Plan
-> Apply Small Change
-> Run Verification
   -> Pass：Update Progress Note -> Next Change
   -> Fail：Record Issue -> Diagnose -> Fix/rollback -> Run Verification
-> Final Verification
-> Consolidate Notes
-> End
```

## 习题 9.4：长时程任务管理与断点续传

在9.6节的“长时程任务管理”案例中，我们看到了上下文工程在实际应用中的价值。请深入分析：

- 案例中使用了“分层上下文管理”策略：即时访问（TerminalTool）+ 会话记忆（MemoryTool）+ 持久笔记（NoteTool）。请分析：这三层之间应该如何协调？什么信息应该放在哪一层？如何避免信息冗余和不一致？
- 假设任务执行过程中发生了中断（如系统崩溃、网络断开），智能体需要从笔记中恢复状态并继续执行。请设计一个“断点续传”机制：如何在笔记中记录足够的状态信息？如何验证恢复后的状态是否正确？
- 长时程任务往往涉及多个子任务的并行或串行执行。请设计一个“任务依赖管理”系统：能够表达任务之间的依赖关系（如“任务B必须在任务A完成后执行”），并自动调度任务执行顺序。这个系统应该如何与NoteTool集成？

### 参考答案

三层上下文应按生命周期和权威性分工。`TerminalTool` 提供即时访问，适合读取当前真实状态，例如文件内容、测试输出、git diff、目录结构。它是事实来源，但不适合长期保存全部输出。`MemoryTool` 保存会话级记忆，适合记录用户偏好、近期交互和当前任务上下文。`NoteTool` 保存持久笔记，适合记录经过整理的计划、决策、进度和恢复点。

避免冗余和不一致的原则是：事实以当前文件系统和命令输出为准；笔记保存摘要、决策和引用；记忆保存对话和偏好。不要把完整文件复制进笔记，只记录路径、行号、摘要和原因。每次关键操作后，笔记应更新“当前状态”，而不是追加大量重复描述。如果发现笔记与真实文件不一致，应以 TerminalTool 重新验证并修正笔记。

断点续传机制需要在笔记中记录足够的恢复状态：

```yaml
task_id: refactor-auth-module
goal: 拆分认证模块并保持测试通过
status: in_progress
current_step: 3
completed_steps:
  - 读取项目结构
  - 提取 AuthService 接口
pending_steps:
  - 更新调用方
  - 运行认证相关测试
modified_files:
  - src/auth/service.ts
  - src/auth/types.ts
last_verified_commit: abc123
last_test_command: npm test -- auth
last_test_result: failed
known_issues:
  - LoginController 仍引用旧方法名
next_action: 修复 LoginController 调用
```

恢复时不应盲目信任笔记，而要执行验证：检查 git status，确认文件是否存在，读取修改文件的当前内容，运行或至少定位上次失败测试，比较笔记中的 `modified_files` 与实际 diff。如果状态匹配，再继续执行；如果不匹配，先更新笔记并重新制定下一步。

任务依赖管理系统可以用 DAG 表达子任务。每个任务包含 `id`、`description`、`depends_on`、`status`、`owner`、`artifacts`、`verification` 和 `notes`。调度器每次选择依赖已完成且状态为 pending 的任务执行。

示例：

```yaml
tasks:
  - id: A
    title: 提取公共接口
    depends_on: []
    status: done
  - id: B
    title: 更新调用方
    depends_on: [A]
    status: pending
  - id: C
    title: 删除旧实现
    depends_on: [B]
    status: blocked
```

与 NoteTool 集成时，项目笔记保存任务图和全局约束，任务笔记保存每个子任务的执行日志、状态和验证结果。调度器每完成一个子任务就更新 NoteTool；中断恢复时从任务图中找出下一个可执行任务。对于并行任务，只有在它们修改文件不冲突、依赖独立时才并行执行；否则应串行化，避免状态冲突。

## 习题 9.5：渐进式披露与探索引导

本章多次提到“渐进式披露”（progressive disclosure）的概念。请思考：

- 在9.2.2节中，渐进式披露被描述为“每一步交互都会产生新的上下文，反过来指导下一步决策”。请设计一个具体的应用场景（如学术论文写作、复杂问题调试），展示渐进式披露如何帮助智能体更高效地完成任务。
- 渐进式披露的一个潜在风险是“探索效率低下”：智能体可能会在不重要的细节上浪费时间，或者错过关键信息。请设计一个“探索引导”机制：通过启发式规则或元认知策略，帮助智能体更聪明地决定“下一步应该探索什么”。
- 对比“渐进式披露”与传统的“一次性加载所有上下文”：在什么类型的任务中，前者有明显优势？在什么类型的任务中，后者可能更合适？请给出至少3个不同类型的任务示例。

### 参考答案

以复杂问题调试为例，用户报告“支付偶发失败”。一次性加载所有支付、订单、用户、库存、日志文件会造成大量噪声。渐进式披露的做法是：先读取错误现象和最近变更；再查看失败日志；根据日志中的错误码定位支付回调模块；发现只在重试场景失败后，再读取幂等逻辑和数据库约束；最后运行相关测试复现。每一步产生的新信息都会指导下一步探索范围，使智能体逐渐缩小问题空间。

流程如下：

```text
症状描述
-> 最近变更和错误日志
-> 定位可疑模块
-> 读取关键函数和调用链
-> 构造复现用例
-> 验证根因
-> 最小修复
```

探索引导机制可以用“信息增益优先”原则。每一步探索前，智能体先写出当前假设、缺失信息和下一步查询能排除什么。优先选择能最大幅度缩小问题空间、成本低、风险低的信息源。启发式规则包括：先看错误日志再看代码；先看入口和变更 diff 再看全仓；先验证假设再扩大范围；如果连续两步没有新信息，切换策略；如果发现高置信根因，先做最小复现。

可以设计元认知检查：

```text
当前目标是什么？
当前最可能的 3 个假设是什么？
下一步读取的信息能验证或排除哪个假设？
这一步成本是多少？
如果没有发现新信息，备用路径是什么？
```

渐进式披露明显适合大型、开放、长时程、信息不确定的任务，例如大型代码库调试、学术论文写作、企业知识库问答、复杂数据分析、法律案情梳理。它能降低上下文噪声，按需获取证据。

一次性加载所有上下文更适合上下文规模小、边界清晰、全局一致性重要的任务。例如审阅一份 2 页合同、总结一篇短论文、检查一个 100 行脚本、比较两个短版本文案。这些任务中，完整上下文不大，全部加载能避免漏看信息。

三个示例对比：

```text
大型代码库 Bug 定位：渐进式披露更好，因为需要按日志和调用链逐步定位。
短文润色：一次性加载更好，因为全文风格一致性重要且内容不长。
学术综述写作：渐进式披露更好，先确定主题，再检索文献，再逐步扩展相关工作。
财务报表单页解释：一次性加载更好，表格完整但规模有限。
多系统故障排查：渐进式披露更好，需要按监控、日志、链路逐层深入。
```
